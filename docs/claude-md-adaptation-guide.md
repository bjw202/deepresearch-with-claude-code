# CLAUDE.md 구조 활용 가이드

이 문서는 현재 리서치 전용으로 최적화된 CLAUDE.md의 구조를 분석하고, 다른 용도(설계, 문서 작성, 분석, 구현)의 프로젝트에서 어떻게 변환하여 사용할 수 있는지 안내한다.

---

## 1. 현재 CLAUDE.md 구조 분석

### 범용 뼈대 (모든 모드에서 유지)

현재 CLAUDE.md는 크게 두 층으로 나뉜다. 아래는 **어떤 프로젝트에서든 그대로 가져갈 수 있는 범용 구조**다.

| 섹션 | 역할 | 범용성 |
| --- | --- | --- |
| **Output Structure** | 산출물 디렉토리 구조, 네이밍 컨벤션 | 🔄 구조는 유지, 폴더명/파일명 변경 |
| **Sub-Agent Permissions** | 서브 에이전트의 파일 쓰기 권한 | ✅ 그대로 유지 |
| **Multi-Agent Policy** | 메인/서브 역할 분리, 품질 우선 원칙 | ✅ 그대로 유지 |
| **Question Expansion** | 착수 전 문제 정의 확장 | ✅ 그대로 유지 |
| **메인 직접 처리 예외** | 서브 에이전트 불필요한 경우 판단 기준 | ✅ 그대로 유지 |
| **Multi-Agent Shared Convention** | 에이전트 간 용어/포맷 통일 | ✅ 그대로 유지 |
| **Journal Policy** | 세션 기록, 의사결정 로그 | ✅ 그대로 유지 |
| **Design-First Policy** | 구현 전 설계 논의 의무화 | ✅ 그대로 유지 |
| **Requirement Validation** | 요청 재검토 | ✅ 그대로 유지 |
| **Final Integration Policy** | 통합 품질 체크리스트 | 🔄 체크리스트 항목 변경 |
| **Failure Prevention** | 안티패턴 목록 | 🔄 모드별 안티패턴 추가 |
| **Search Tool Usage** | 검색 도구 사용법 | 🔄 필요 시 도구 교체 |

### 리서치 특화 (변환 시 교체 대상)

| 섹션 | 리서치에서의 역할 | 변환 시 |
| --- | --- | --- |
| **Required Sub-Agents** (Researcher, Critic, Verifier, Synthesizer, Journal) | 리서치 관점별 분업 | 모드별 역할로 교체 |
| **Researcher 사고 지침** 6가지 | 출처 평가, 반증 탐색 등 | 모드별 사고 지침으로 교체 |
| **Synthesis 필수 섹션** 5가지 | 근거 신뢰도, 상충점 등 | 모드별 통합 섹션으로 교체 |
| **Search Tool Usage** 단계별 흐름 | 조사 깊이 단계 (0\~3단계) | 모드별 도구 흐름으로 교체 |
| **Research-to-Presentation Pipeline** | 리서치 → 발표자료 연결 | 모드별 후속 파이프라인으로 교체 |

## 2. 변환 템플릿

### 변환 절차

1. CLAUDE.md를 복사한다
2. **범용 뼈대**는 유지한다 (위 표의 ✅ 항목)
3. **🔄 항목**을 아래 모드별 가이드에 따라 교체한다
4. 프로젝트 고유 도구/스크립트가 있으면 Search Tool Usage를 해당 도구로 교체한다

### 공통 변환 규칙

모든 모드에 공통으로 적용되는 변환 규칙:

```
## Output Structure 변환

docs/
├── {mode-output}/              # research/ → 모드별 산출물 폴더명
│   └── {date}-{topic}/
│       ├── 00-synthesis.md     # 종합 문서 (구조 동일)
│       ├── 01-{관점1}.md
│       └── 99-{부속자료}.md
└── dev-journal/                # 저널 구조는 동일
```

## 3. 모드별 변환 가이드

---

### Mode A: Design (설계)

시스템 아키텍처, API 설계, 데이터 모델링 등 설계 작업에 사용한다.

#### 산출물 구조

```
docs/
├── design/
│   └── {date}-{topic}/
│       ├── 00-design-decision.md    # 설계 결정 종합 (여기서 시작)
│       ├── 01-{방안A}.md            # 설계 방안별 상세
│       ├── 02-{방안B}.md
│       ├── 03-constraints.md        # 제약 조건 분석
│       └── 99-trade-off-review.md   # 트레이드오프 리뷰
└── dev-journal/
```

#### 서브 에이전트 역할

| 역할 | 담당 | 리서치 모드 대응 |
| --- | --- | --- |
| **Architect** | 설계 방안 제안, 구조 정의 | Researcher |
| **Constraint Analyst** | 기술적/비즈니스 제약 분석, 실현 가능성 평가 | Researcher (다른 관점) |
| **Critic** | 설계 허점, 확장성 문제, 숨은 복잡도 탐지 | Critic (동일) |
| **Prototyper** | 핵심 불확실성의 PoC 코드 작성 | (리서치에는 없음) |
| **Journal** | 설계 의사결정 로그 | Journal (동일) |

Architect가 2명 이상이면 Critic은 필수. 이 규칙은 리서치의 "Researcher 2명 이상 → Critic 필수"와 동일하다.

#### Architect 사고 지침

```
1. 제약 우선: 기술적/비즈니스 제약을 먼저 나열하고, 제약 안에서 설계한다.
   제약이 불명확하면 [가정: {내용}] 태깅 + 가정이 틀릴 때의 영향 한 문장 기술
2. 대안 비교: 최소 2개 방안을 비교한다. 단일 방안만 제시할 경우 "대안 불필요 사유" 명시
3. 실패 모드: 각 설계 방안의 실패 시나리오 1~2개 + 완화 전략
4. 진화 경로: "이 설계가 6개월 후 어떤 방향으로 변경될 가능성이 높은가" 한 문장
5. 의존성 명시: 외부 시스템, 라이브러리, 팀 간 의존성을 명시적으로 나열
6. 단순성 검증: "이보다 더 단순한 방법이 있는가?" 자문. 있으면 그것을 방안 1로 제시
```

#### Critic 체크리스트 (설계 모드)

1. 과잉 설계: 현재 요구사항 대비 불필요한 추상화가 있는가
2. 숨은 복잡도: 명시되지 않은 엣지 케이스, 동시성 문제, 데이터 정합성 위험이 있는가
3. 서브에이전트 간 상충점: Architect 산출물 간 전제나 결론이 모순되는가
4. 운영 관점: 배포, 모니터링, 롤백이 고려되었는가
5. 확신도 교정: PoC 없이 "잘 동작할 것"이라는 가정이 있는가
6. 문제 정의 검토: 설계 대상 자체가 올바른가. 더 좋은 문제 정의가 있는가

#### Synthesis 필수 섹션 (설계 모드)

1. **설계 결정 매트릭스**: 방안별 장단점, 제약 충족도, 리스크 (표)
2. **트레이드오프 테이블**: 성능 vs 단순성, 유연성 vs 명확성 등 핵심 트레이드오프 (표)
3. **결정 근거**: 최종 선택 방안 + 선택 이유 + 기각된 대안의 기각 사유
4. **실행 계획**: 구현 순서, 마일스톤, 검증 방법
5. **후속 설계 질문**: 이번에 결정하지 못한 열린 질문 2\~3개

### Mode B: Writing (문서 작성)

기술 문서, 백서, 제안서, 튜토리얼 등 긴 글쓰기 작업에 사용한다.

#### 산출물 구조

```
docs/
├── writing/
│   └── {date}-{topic}/
│       ├── 00-final-draft.md        # 최종 원고 (여기서 시작)
│       ├── 01-{섹션1}-draft.md      # 섹션별 초안
│       ├── 02-{섹션2}-draft.md
│       ├── 03-references.md         # 참고 자료 목록
│       └── 99-editorial-review.md   # 편집 리뷰
└── dev-journal/
```

#### 서브 에이전트 역할

| 역할 | 담당 | 리서치 모드 대응 |
| --- | --- | --- |
| **Writer** | 섹션별 초안 작성 | Researcher |
| **Fact-Checker** | 인용, 수치, 기술적 정확성 검증 | Verifier |
| **Editor** | 구조, 흐름, 일관성, 중복 검토 | Critic |
| **Audience Analyst** | 독자 수준 적합성, 용어 난이도 평가 | (리서치에는 없음) |
| **Journal** | 원고 버전 이력, 편집 결정 로그 | Journal (동일) |

Writer가 2명 이상이면 Editor는 필수.

#### Writer 사고 지침

```
1. 독자 정의: 글의 대상 독자를 명시하고, 해당 독자의 사전 지식 수준에 맞춘다.
   독자가 불명확하면 [가정 독자: {수준}] 태깅
2. 주장-근거 연결: 모든 핵심 주장에 근거를 연결한다. 근거 없으면 "저자 견해" 명시
3. 구체성: 추상적 서술 대신 예시, 수치, 사례를 우선한다. "많은" → "약 70%"
4. 흐름 연결: 각 섹션의 끝에서 다음 섹션으로의 논리적 연결을 한 문장으로 작성
5. 반대 독자: "이 글에 동의하지 않을 독자는 어떤 점을 지적할 것인가" 1개 제시
6. 한 줄 요약: 섹션 작성 후 "이 섹션의 핵심을 한 문장으로" — 쓸 수 없으면 구조 문제
```

#### Editor 체크리스트 (문서 모드)

1. 구조적 완결성: 서론-본론-결론 흐름이 논리적인가
2. 일관성: 용어, 표기법, 어조가 섹션 간 통일되어 있는가
3. Writer 간 상충점: 동일 개념에 대해 상충하는 서술이 있는가
4. 중복: 동일 내용이 여러 섹션에 반복되는가
5. 근거 강도: 핵심 주장의 뒷받침이 충분한가. 출처가 명시되었는가
6. 독자 적합성: 대상 독자의 수준에 비해 너무 어렵거나 너무 쉬운 부분이 있는가

#### Synthesis 필수 섹션 (문서 모드)

1. **원고 구조 맵**: 전체 흐름을 한 눈에 보여주는 섹션별 요약 (표)
2. **편집 이슈 테이블**: 발견된 문제, 수정 방안, 처리 상태 (표)
3. **용어 통일표**: 문서 전체에서 사용하는 핵심 용어의 정의와 표기법
4. **독자 적합성 평가**: 대상 독자 기준 난이도, 개선 필요 부분
5. **후속 작업**: 추가 검토가 필요한 섹션, 확인이 필요한 사실 관계

### Mode C: Analysis (분석)

데이터 분석, 비즈니스 분석, 기술 평가, 비교 분석 등에 사용한다.

#### 산출물 구조

```
docs/
├── analysis/
│   └── {date}-{topic}/
│       ├── 00-findings.md           # 분석 결과 종합 (여기서 시작)
│       ├── 01-{분석관점1}.md        # 관점별 상세 분석
│       ├── 02-{분석관점2}.md
│       ├── 03-data-notes.md         # 데이터 출처/한계 기록
│       └── 99-methodology-review.md # 방법론 검토
└── dev-journal/
```

#### 서브 에이전트 역할

| 역할 | 담당 | 리서치 모드 대응 |
| --- | --- | --- |
| **Analyst** | 관점별 데이터 분석, 패턴 발견 | Researcher |
| **Methodologist** | 분석 방법론 적절성 검증, 통계적 타당성 | Verifier |
| **Critic** | 편향, 누락 변수, 과적합, 인과-상관 혼동 탐지 | Critic (동일) |
| **Recommender** | 분석 결과 → 실행 가능한 권고안 도출 | (리서치에는 없음) |
| **Journal** | 분석 과정 기록, 가정 변경 로그 | Journal (동일) |

Analyst가 2명 이상이면 Critic은 필수.

#### Analyst 사고 지침

```
1. 데이터 출처: 사용하는 데이터의 출처, 수집 시점, 한계를 명시한다.
   2차 데이터면 [2차 데이터: {원출처}] 태깅 + 원데이터와의 차이 한 문장
2. 대안 해석: 동일 데이터에서 가능한 다른 해석을 최소 1개 제시. 없으면 "대안 해석 없음" 명시
3. 통계적 유의성: 수치 비교 시 차이가 유의미한지 판단 근거 제시. 판단 불가면 "유의성 미검증" 명시
4. 실행 연결: "이 분석 결과로 어떤 의사결정이 가능한가"까지 연결
5. 숨은 변수: 분석에 포함되지 않았지만 결과에 영향을 줄 수 있는 변수 1~2개 제시
6. 재현성: 동일 분석을 반복했을 때 같은 결론이 나올 조건을 명시
```

#### Critic 체크리스트 (분석 모드)

1. 선택 편향: 분석 대상 데이터가 모집단을 대표하는가
2. 인과 추론: 상관관계를 인과관계로 잘못 해석하고 있지 않은가
3. Analyst 간 상충점: 동일 데이터에 대해 상반된 결론이 있는가
4. 누락 변수: 결과를 뒤집을 수 있는 미고려 변수가 있는가
5. 확신도 교정: 데이터 양/질 대비 과도한 확신이 있는가
6. 분석 범위: 분석 프레임이 원래 질문에 적절한가. 더 적합한 분석 프레임이 있는가

#### Synthesis 필수 섹션 (분석 모드)

1. **발견 신뢰도 매트릭스**: 핵심 발견별 데이터 출처, 표본 크기, 확신도, 재검증 필요 여부 (표)
2. **상충 분석 테이블**: Analyst 간 상반된 결론 + 판단 근거 (표, 해당 시)
3. **의사결정 가이드**: "지표가 X이면 → Y를 조정하라" 형태의 실행 가이드
4. **한계 및 주의사항**: 데이터 한계, 일반화 불가 범위, 추가 데이터 필요 영역
5. **후속 분석 질문**: 이번에 답하지 못한 질문, 추가 데이터로 확인해야 할 것 2\~3개

### Mode D: Implementation (구현)

코드 작성, 리팩토링, 마이그레이션 등 실제 구현 작업에 사용한다.

#### 산출물 구조

```
docs/
├── implementation/
│   └── {date}-{topic}/
│       ├── 00-implementation-plan.md  # 구현 계획 종합 (여기서 시작)
│       ├── 01-{모듈1}-spec.md         # 모듈별 구현 스펙
│       ├── 02-{모듈2}-spec.md
│       ├── 03-test-strategy.md        # 테스트 전략
│       └── 99-code-review.md          # 코드 리뷰
└── dev-journal/
```

실제 코드는 `docs/` 밖 프로젝트 소스 디렉토리에 작성한다. `docs/implementation/`에는 계획과 스펙만 저장한다.

#### 서브 에이전트 역할

| 역할 | 담당 | 리서치 모드 대응 |
| --- | --- | --- |
| **Implementer** | 모듈별 코드 작성 | Researcher |
| **Tester** | 테스트 코드 작성, 엣지 케이스 발견 | Verifier |
| **Reviewer** | 코드 품질, 보안, 성능 리뷰 | Critic |
| **Integrator** | 모듈 간 통합, 인터페이스 정합성 확인 | Synthesizer |
| **Journal** | 구현 결정 로그, 기술 부채 기록 | Journal (동일) |

Implementer가 2명 이상이면 Reviewer는 필수.

#### Implementer 사고 지침

```
1. 스펙 확인: 구현 전에 스펙/인터페이스가 확정되었는지 확인.
   불명확하면 [가정: {내용}] 태깅 + 가정이 틀릴 때의 수정 범위 한 문장
2. 최소 구현: 요구사항을 충족하는 가장 단순한 구현부터 시작. 최적화는 측정 후에
3. 에러 처리: 외부 입력과 시스템 경계에서만 방어적 코드. 내부 호출은 신뢰
4. 테스트 연결: "이 코드를 어떻게 테스트할 것인가"를 구현 전에 결정
5. 의존성 최소화: 새 라이브러리 추가 전 "기존 도구로 가능한가" 확인. 불가능할 때만 추가
6. 되돌림 가능성: "이 변경을 쉽게 되돌릴 수 있는가" 자문. 불가능하면 단계적 적용 고려
```

#### Reviewer 체크리스트 (구현 모드)

1. 보안: OWASP Top 10 취약점이 있는가 (인젝션, XSS, 인증 우회 등)
2. 성능: 명백한 N+1 쿼리, 불필요한 반복, 메모리 누수가 있는가
3. Implementer 간 상충점: 모듈 간 인터페이스 불일치, 중복 구현이 있는가
4. 에러 처리: 실패 시 시스템이 안전한 상태로 돌아가는가
5. 테스트 커버리지: 핵심 경로와 엣지 케이스가 테스트되었는가
6. 단순성: 더 단순한 구현이 가능한가. 과잉 추상화가 있는가

#### Synthesis 필수 섹션 (구현 모드)

1. **모듈 상태 매트릭스**: 모듈별 구현 상태, 테스트 상태, 리뷰 상태 (표)
2. **통합 이슈 테이블**: 모듈 간 인터페이스 문제, 해결 방안 (표, 해당 시)
3. **배포 체크리스트**: 배포 전 확인 항목, 롤백 계획
4. **기술 부채 기록**: 의도적으로 미루거나 타협한 사항과 그 이유
5. **후속 작업**: 다음 스프린트에서 처리할 개선 사항 2\~3개

## 4. 변환 시 핵심 원칙 (반드시 유지)

아래 원칙은 모드와 무관하게 CLAUDE.md의 핵심 가치를 구성한다. 변환 시 절대 제거하지 않는다.

### 4.1 품질 &gt; 속도

> "품질 우선은 느려도 된다는 뜻이지, 불필요하게 비효율적이어도 된다는 뜻은 아니다."

모든 모드에서 이 원칙을 유지한다. 에이전트 수를 늘리는 것 자체가 목적이 아니며, 분업이 품질에 기여할 때만 멀티에이전트를 사용한다.

### 4.2 Question Expansion

착수 전 문제 정의를 확장하는 습관은 리서치뿐 아니라 설계, 분석, 구현 모두에서 유효하다. 4가지 항목(핵심 질문, 전제 검증, 인접 질문, 반대 시나리오)을 그대로 유지한다.

### 4.3 Critic/검증자 의무화

핵심 작업자가 2명 이상이면 반드시 검증 역할을 둔다. 역할 이름은 달라도(Critic, Editor, Reviewer) 기능은 동일하다: **상충점 발견, 품질 검증, 확신도 교정**.

### 4.4 메인 에이전트 = 오케스트레이터

메인은 직접 작업하지 않고 조직·통합한다. 이 원칙을 깨면 병목이 생기고 품질이 떨어진다.

### 4.5 산출물 = 폴더 단위

하나의 작업 세션은 하나의 폴더에 모든 산출물을 보관한다. Loose 파일을 루트에 놓지 않는다.

### 4.6 Final Integration ≠ 단순 병합

서브 에이전트 결과를 그대로 이어붙이지 않는다. 중복 제거, 상충점 정리, 우선순위화를 거친다.

### 4.7 저널 기록

의미 있는 세션에서는 반드시 저널을 남긴다. 의사결정 이유가 없는 기록은 기록이 아니다.

## 6. 모드별 Failure Prevention 추가 항목

리서치 모드의 기본 안티패턴에 더해, 각 모드에서 추가로 경계해야 할 항목:

### Design

- PoC 없이 "잘 동작할 것"이라는 가정에 기반한 설계 확정
- 현재 요구사항에 비해 과도한 추상화/확장성 설계
- 운영(배포, 모니터링, 롤백)을 고려하지 않은 설계

### Writing

- 근거 없이 단정적 어조로 작성된 핵심 주장
- 독자 수준을 정의하지 않고 작성하여 난이도가 불균일한 문서
- 섹션 간 용어/표기법 불일치

### Analysis

- 상관관계를 인과관계로 제시
- 데이터 한계를 명시하지 않은 채 일반화
- 결론에 유리한 데이터만 선별(체리피킹)

### Implementation

- 테스트 전략 없이 구현 착수
- 모듈 간 인터페이스를 합의하지 않고 병렬 구현
- 기술 부채를 기록하지 않고 "나중에 고치겠다"는 구두 합의만 남김
