- **선행 리서치**: `docs/research/2026-03-18-build123d-gear-system/` (build123d 기어 시스템)

## 사용자 요청 요약

코딩 에이전트(Claude Code 등)로 build123d 3D 모델을 시행착오 최소화하여 제작하는 방법을 연구.

핵심 의문:

1. 시각 모델을 만드는데 코딩 에이전트가 결과를 볼 수 없다 — 어떻게 검증?
2. 의도를 자연어로 잘 전달할 수 있는가?
3. 코드는 잘 짜는가? 오류가 있으면 어떻게 수정?
4. 바이브코딩이 3D CAD에서 작동하는가?

## 문제 정의

텍스트 코드는 실행 결과(출력, 에러)가 텍스트이므로 코딩 에이전트가 직접 읽고 디버깅할 수 있다. 그러나 3D CAD 코드는:

- 실행 에러는 텍스트로 잡을 수 있지만
- 시각적 오류(형상이 의도와 다름)는 텍스트만으로 감지 불가
- 코딩 에이전트가 OCP CAD Viewer 렌더링을 볼 수 없음

이 \*\*"시각 피드백 갭"\*\*을 어떻게 메우는가가 핵심 과제.

## 핵심 가정

1. LLM은 build123d 코드를 어느 정도 생성할 수 있다 (CadQuery 학습 데이터 존재, build123d는 적음)
2. 코드 실행 에러는 기존 디버깅 루프로 해결 가능
3. 시각적 오류는 수치 검증(체적, bounding box) + VLM + 사용자 피드백의 조합으로 완화 가능
4. 단계적 빌드가 한 번에 전체 생성보다 성공률이 높을 것

## 핵심 결론

1. **바이브코딩은 "부분적으로 작동"** — 에이전트 70-80% + 사용자 20-30%
2. **시각 피드백 갭 해결**: 수치 검증(A) + 단계적 빌드(D)가 가장 실용적
3. **Skill Card가 핵심** — user message에 코드 템플릿 주입 (Henqo 발견)
4. **현존 도구**: VibeCAD, ocp-viewer-mcp, cad-agent
5. **어셈블리는 미해결 과제**

## Rejected Alternatives

- **"직접 build123d + Claude Code로 실험"** → 먼저 현황 파악 후 실험이 효율적
- **"CadQuery 기반으로 연구"** → build123d가 LLM에 더 구조적 이점 (이전 리서치 결론)
