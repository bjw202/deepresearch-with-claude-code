## 2. 수치 근거 검증

**판정: 부분적 우려**

### 2.1 인벌류트 파라메트릭 방정식 — 정확

R1 §1.1과 R2 §2.2의 인벌류트 방정식은 동일하며(파라미터 명칭만 $t$ vs $\phi$ 차이), 교과서급 표준식이다. 교차 검증 충분.

### 2.2 전위 기어 수식 — R1과 R3 간 일관성 확인 필요

| 항목 | R1 | R3 |
|------|----|----|
| 피치원 이두께 | $s_0 = m(\pi/2 + 2x\tan\alpha_0)$ | $s = \pi m/2 + 2xm\tan\alpha$ |
| 최소 전위 계수 | $x_{min} = 1 - z/17$ | $x_{min} = (17 - N)/17$ |
| 언더컷 최소 이수 | $z_{min} = 2/\sin^2(\alpha_0)$ | $N_{min} = 2h_a^*/\sin^2\alpha$ |

이두께 수식: R1의 $s_0 = m(\pi/2 + 2x\tan\alpha_0)$과 R3의 $s = \pi m/2 + 2xm\tan\alpha$는 **수학적으로 동일**하다($m$을 묶는 방식의 차이). 그러나 R3에서 $m$이 분배되지 않은 형태로 적혀 있어 초보 독자에게 혼동을 줄 수 있다.

최소 전위 계수: R1은 $x_{min} = 1 - z/17$, R3은 $x_{min} = (17 - N)/17$. **동일한 수식**이지만 변수명이 $z$ vs $N$으로 다르다.

언더컷 조건: R1은 일반형 $z_{min} = 2/\sin^2(\alpha_0)$, R3은 $h_a^*$를 명시한 형태. R3이 더 일반적이다($h_a^* = 1$일 때 R1과 일치). **일관성은 유지되나 표기법 통일이 필요**.

### 2.3 베벨 기어 근사법 한계 — 부분적으로 기술

R2 §4.3에서 "Tredgold 근사는 이폭이 크거나(b > 0.3R) 큰 압력각에서 오차가 증가한다"고 언급. 그러나 **오차의 정량적 크기**(예: 이폭/원뿔거리 비율별 각도 오차)가 없다. KU Leuven 논문(★★★)이 출처에 있으므로 정량 데이터 추출이 가능했을 것.

### 2.4 호브 3D 좌표 수식 — 교차 검증 어려움

R3 §1.4의 호브 3D 파라메트릭 방정식은 ★★☆로 표기. 나선 배치(helix wrapping) 수식의 $\sin\gamma_o$/$\cos\gamma_o$ 항이 Bergseth(2011) 원문과 정확히 대응하는지 확인이 필요하다. 이 수식은 단일 출처(KTH 석사논문) 의존이므로 구현 전 원문 대조가 권장된다.

### 2.5 피니언 커터 좌표 변환 행렬 — 오류 의심

R3 §2.3에서 좌표 변환 행렬 $\mathbf{M}_{gc}$가 **두 번 제시**된다. 첫 번째 행렬(line 200-204)과 두 번째 "정확한 형태"(line 208-212) 사이에 차이가 있다. 첫 번째 행렬의 (1,2) 요소에 `r_c(1 - cos φ_g) · (r_c/r_g) · r_g`라는 의미 불명의 항이 있다. **첫 번째 행렬은 삭제하거나 "근사 형태"로 명확히 표기해야** 한다. 현재 상태로는 구현자가 어떤 행렬을 사용해야 하는지 혼동된다.

## 3. 서브에이전트 간 상충점

**판정: 다수의 표기 불일치 발견, 수학적 모순은 제한적**

### 3.1 파라미터 기호 불일치 (가장 심각)

| 개념 | R1 | R2 | R3 |
|------|----|----|-----|
| 이수 | $z$ | $z$ | $N$, $N_g$, $N_c$ |
| 인벌류트 파라미터 | $t$ | $\phi$ | $\delta$ |
| 압력각 | $\alpha_0$ | $\alpha$, $\alpha_0$ | $\alpha$, $\alpha_n$ |
| 인벌류트 함수 표기 | $\text{inv}(\alpha)$ | $\text{inv}(\alpha)$ | $\text{invol}(\alpha)$ |
| 필릿 반경 기호 | $r_c$ (랙 커터 팁) | $\rho_f$ (근사치) | $r_f$, $r_{f0}$ |
| 피치원 반지름 | $r_0$ | $r$ | $r$ |

**가장 위험한 불일치**: 인벌류트 함수를 R1/R2는 `inv()`, R3은 `invol()`로 표기. 같은 함수($\tan\alpha - \alpha$)인데 이름이 다르면 코드 구현 시 별도 함수로 오인할 수 있다.

**이수 기호**: R1/R2는 $z$, R3은 $N$을 사용. 기어 공학에서 두 표기 모두 통용되지만(유럽/ISO는 $z$, AGMA는 $N$), **한 프로젝트 내에서는 통일이 필수**.

### 3.2 트로코이드 필릿 수식 상충

이것이 **가장 심각한 상충**이다:

- **R1 §3.3**: 랙 커터 팁의 트로코이드. 변수 $u$(랙 절삭 위치), $r_c \approx 0.38m$, $\varphi(u)$의 정의 불명확.
- **R2 §2.3**: 호브 가공 트로코이드. 에피트로코이드형 수식 사용. 변수 $r_c$(공구 생성원 반지름), $d_c$(공구 끝 거리). **이 $r_c$가 R1의 $r_c$(랙 커터 팁 반경)와 다른 의미**.
- **R3 §2.5**: Fetvaci 기반 에피트로코이드. 1차/2차 트로코이드를 구분. 가장 정밀한 표현.

**결론**: 세 보고서가 각각 다른 수준의 필릿 모델을 제시하면서 같은 기호($r_c$)를 다른 의미로 사용. 통합 시 R3의 Fetvaci 방법을 기준으로 삼고, R1/R2의 필릿 수식은 "간략 근사" 레이블을 붙여야 한다.

### 3.3 R1의 치형 생성과 R3의 공구 치형 생성 일관성

R1 §3.4의 알고리즘은 **기하학적 인벌류트 직접 생성** 방식이다(기초원에서 인벌류트 점을 직접 계산). R3의 공구 방식은 **포락선/창성 운동**으로 치형을 생성한다. 두 방법은 이론적으로 동일한 인벌류트 플랭크를 생성해야 하지만, **필릿 영역에서 결과가 다르다**. R1은 원호 근사 필릿, R3은 트로코이드 필릿. 이 차이가 명시적으로 서술되지 않았다.

### 3.4 이끝원/이뿌리원 수식 — R2의 전위 미포함

R2 §2.1의 기본 파라미터 표에서:
- $r_a = r + m$, $r_f = r - 1.25m$

이는 전위 계수 $x = 0$인 표준 기어에만 해당. R1은 $d_a = m(z + 2 + 2x)$로 전위를 포함. R2가 표준 기어만 다루는 것이라면 명시해야 하고, 전위 기어도 포함한다면 수식 보정이 필요하다.

## 4. 누락 관점

### 4.1 수치 구현 엣지 케이스 — 부분 누락

**다뤄진 것**:
- R1 §5.1: 기초원 아래 인벌류트($r_f < r_b$) 처리 ✓
- R1 §5.1: inv() 역함수 뉴턴법 ✓
- R3 §7.1: 포락선 특이점, 소각도 테일러 전개 ✓

**누락된 것**:
- **이수 극단값**: $z = 1$ 또는 $z = 2$에서 인벌류트가 성립하는가? 수학적으로 가능하지만 실용적으로 무의미한 경우의 하한 검증.
- **전위 계수 극단값**: $x > 1.0$ 또는 $x < -0.5$에서 이끝이 뾰족해지거나(pointed tooth) 이뿌리원이 기초원보다 커지는 경우의 처리.
- **부동소수점 정밀도**: `tan(alpha)`이 $\alpha \to \pi/2$에서 발산. 압력각이 비표준으로 큰 경우(25+)에서 수치 오버플로 가능성.
- **뉴턴법 수렴 실패**: R1과 R2 모두 뉴턴법 초기값을 $x^{1/3}$으로 설정하지만, 수렴 실패 시 fallback 전략이 없다.

### 4.2 좌표계 통일 — 부분적으로 다뤄짐

R2 §8.1에서 좌표계 컨벤션을 명시(기어 축 = z축, 중심 = 원점). 그러나:
- R3의 호브/피니언 커터 좌표계($S_h$, $S_c$, $S_g$)가 R2의 컨벤션과 어떻게 대응하는지 기술 없음.
- **베벨 기어의 좌표계**: R2 §4.4에서 "원뿔축을 z축으로 가정"이라고 하지만, 스퍼/헬리컬과 베벨이 같은 좌표계를 공유하는지 불명확.

### 4.3 단위계 — 거의 누락

- R1에서 "모든 각도는 라디안"이라고 명시 ✓
- 그러나 **mm vs inch 문제는 전혀 다루지 않았다**. AGMA는 Diametral Pitch(inch계), ISO/KHK는 Module(mm계)을 사용. 시뮬레이션 앱에서 두 시스템을 지원할 경우의 변환 수식이 없다:
  - $m = 25.4 / P_d$ (Diametral Pitch → Module)
  - $p = \pi / P_d$ (Diametral Pitch → Circular Pitch)

### 4.4 검증/테스트 방법 — 크게 누락

생성된 기어가 올바른지 확인하는 방법이 거의 없다:
- R3 §5(이두께 측정 수학)이 스팬/오버핀 측정을 다루지만, 이는 물리적 측정 방법.
- **소프트웨어 검증 방법**이 없다:
  - 생성된 인벌류트 점이 실제로 인벌류트 위에 있는지 확인 (역계산으로 기초원 접선 검증)
  - 접촉비가 1.2 이상인지 자동 검증
  - 언더컷/간섭 발생 여부 자동 검출
  - 두 기어 맞물림 시 간섭(tooth overlap) 없음 검증
  - 기존 CAD 소프트웨어(KISSsoft 등) 결과와의 비교 기준

### 4.5 실패 모드 — 부분적

R1 §5.1에서 언더컷, 이끝 간섭을 언급. R2 §5.4에서 내접 기어 간섭 조건 제시. 그러나:
- **무효 파라미터 조합의 체계적 목록**이 없다. 예:
  - $z_1 + z_2$가 너무 작아 기어쌍이 물리적으로 불가능한 경우
  - 내접 기어에서 $z_2 - z_1 < 5$인 경우의 구체적 실패 양상
  - 헬리컬 기어에서 $\beta > 45°$의 실용적 한계
  - 전위 합 $(x_1 + x_2)$의 상한/하한

## 6. 문제 정의 검토

### 6.1 "프로그래밍 구현 가능 수준" 달성도

**인벌류트 스퍼 기어: 90% 달성**. R1의 알고리즘 + R2의 3D 압출 코드로 즉시 구현 가능. 필릿 부분만 추가 정리 필요.

**헬리컬 기어: 80% 달성**. R2 §3.3의 twist 코드가 명확. 단, 전위 헬리컬의 이두께 계산이 R1의 스퍼 전용 수식만으로는 불충분 — 법선면/횡단면 변환 후 적용해야 하는데 이 절차가 명시적이지 않다.

**베벨 기어: 60% 달성**. Tredgold 근사 알고리즘은 있으나, R2 §4.4의 3D 좌표 변환 코드가 **수학적으로 불완전**하다. `scaled_profile[:, 1] + r_pos`라는 단순 덧셈은 원뿔면 투영이 아니라 원통면 오프셋이다. 실제로 원뿔면 위의 정확한 좌표는 구면 좌표 변환이 필요하다.

**내접 기어: 70% 달성**. 수식은 충분하나 "치의 두께와 공간이 반전된다"는 설명이 알고리즘 수준으로 구체화되지 않았다. 어떤 부호를 바꾸는지, 코드에서 어떤 변수를 반전하는지 명시 필요.

**가공 공구 시뮬레이션: 75% 달성**. R3의 포락선 이론과 구현 순서 권고가 우수. 그러나 `envelope_generate()` 함수 본체가 `...`으로 비어 있어 핵심 알고리즘이 빠져 있다.

### 6.2 수식만 있고 알고리즘 흐름이 없는 부분

- **R3 §2.3-2.4**: 창성 운동 + 맞물림 방정식이 수식으로 제시되지만, "파라미터 $\delta$와 $\phi_c$를 어떤 범위에서, 어떤 순서로, 어떻게 스캔하여 기어 치형 좌표를 모으는가"의 알고리즘 흐름이 없다. R1 §3.4 수준의 단계별 의사코드가 필요.
- **R2 §4.3 구면 인벌류트**: 수식만 제시되고 구현 알고리즘 전무. "Litvin의 벡터 형식으로 완전히 표현된다"고만 언급.
- **R3 §4.1 포락선 이산화**: `envelope_generate()` 함수 시그니처만 있고 구현 없음.

### 6.3 추가 정보 필요 여부

이 레퍼런스만으로 **스퍼 기어 프로파일 생성기**는 구현 가능하다. 그러나 다음에는 추가 정보가 필수:

1. **맞물림 시뮬레이션**: 두 기어의 동시 회전 + 접촉점 추적 알고리즘
2. **백래시 모델링**: 치형 법선 방향 오프셋 알고리즘
3. **스파이럴 베벨**: Gleason 시스템 없이는 근사에 그침 (보고서 자체가 인정)
4. **웜 기어**: 세 보고서 모두에서 완전히 누락. 기어 시뮬레이션 앱이라면 주요 기어 유형 중 하나.

### 2. 코드 품질

**판정: 양호 (주의사항 3건)**

R4와 R5의 모든 Python 코드는 완전한 함수 본체를 갖추고 있다. `...`이나 `pass`로 비워둔 함수는 발견되지 않았다. `math`/`numpy` 외 외부 의존성 없음(matplotlib은 선택적). docstring, 타입 힌트, 사용 예시가 일관적으로 포함되어 있다.

**주의사항**:

1. **R4 §D.1 `verify_involute_points()`의 검증 로직 중복**: 라인 820-831에서 `x_reconstructed`, `y_reconstructed`를 계산한 후 `r_reconstructed`와 `r`을 비교하지만, 인벌류트의 정의상 $r = r_b\sqrt{1+t^2}$이므로 이 비교는 항상 0이다. 실제 검증은 라인 838-840의 `expected_r_sq` 비교에서 수행된다. 전자는 중복 코드이며 삭제해도 된다. 기능적 오류는 아니지만 코드 정리가 필요하다.

2. **R4 §F.2 `_points_inside_polygon()` 성능**: Ray casting이 $O(K \times N^2)$ ($K$: 폴리곤 꼭짓점, $N$: 그리드 크기)으로, `grid_size=500` + `n_phi=720`이면 단일 기어 치형 생성에 수십 초 소요될 수 있다. 문서에서 성능 제약을 언급하고 있으나(`n_phi` 상한 5000), 앱 목적에 따라 이 방식이 실용적이지 않을 수 있다. 해석적 방법(§F.1)과의 이중 전략을 §G에서 제안한 것은 적절하다.

3. **R5 `generate_worm_wheel_envelope()`의 포락 근사**: 이 함수는 "모든 phi1에 대한 점 집합의 외곽 = 웜 휠 치면"이라고 설명하지만, 실제로는 점 집합을 단순히 `np.vstack()`으로 합치기만 하고, 포락 조건 방정식(식 29)을 풀지 않는다. 즉 **진정한 포락선이 아닌 점 구름(point cloud)**을 반환한다. 후처리(convex hull, alpha shape 등)로 외곽을 추출해야 실제 치면이 된다. 이 한계가 코드 내에 명시적으로 서술되어 있지 않다.

### 3. 기존 보고서와의 일관성

**판정: 양호**

- **R4**: 문서 앞머리에 "프로젝트 기호 표준"을 명시($z$, $t$, $\alpha_0$, `inv()`, $r_0$)하고 전체에 걸쳐 일관되게 사용. R3의 $N$, `invol()` 표기를 사용하지 않았다.
- **R4 §B.2**: Fetvaci 변환 행렬에서 $N_c$/$N_g$ 표기를 사용하는데, 이는 R3의 AGMA 표기법이다. 프로젝트 표준($z$)과 불일치하지만, `M_gc()` 함수의 파라미터명으로만 사용되고 문맥상 혼동 가능성이 낮다. 다만 **일관성 원칙에 따르면 `z_c`/`z_g`가 더 적절**하다.
- **R5**: $z_1$/$z_2$ (이수), $\alpha_n$/$\alpha_x$ (압력각)을 사용. 웜 기어 도메인에서는 축방향/법선 구분이 필수이므로 $\alpha_x$/$\alpha_n$은 적절하다. 인벌류트 함수(`inv()`)를 직접 사용하는 경우가 없어(웜 기어는 인벌류트 함수를 기본으로 쓰지 않음) 기호 충돌이 없다.

**결론**: R4/R5 모두 프로젝트 표준 기호를 준수한다. R4 §B.2의 `N_c`/`N_g`만 미세한 불일치가 있으나 기능적 문제는 없다.

| ZN | 법선 벡터 정확 형식 제공 (식 23) | **조건부** | $\rho$ 범위와 위상차 계산이 추가로 필요. 확신도 ★☆☆ (Eng-Tips 포럼 단독) |
| ZK | 수식 미제공, 원뿔 숫돌 설명만 | **아니오** | "CAD 소프트웨어의 좌표 변환 파이프라인으로 구현"이라는 설명만. 시뮬레이션 앱 목적에는 부족 |

**판정**: ZA는 완전하다. ZI는 수식은 있으나 코드가 없다. ZN/ZK는 구현 가능 수준에 미달한다. ZA가 산업용으로 가장 보편적이므로 첫 구현 대상으로는 충분하지만, "유형별 수식 완전 제공"이라는 목표에는 **ZI 코드 누락, ZK 수식 부재**가 남는다.

#### 4.2 웜 휠 포락 생성 알고리즘 — 불완전

R5 §3.2에서 포락 생성의 4단계 이론(좌표계 → 변환행렬 → 포락 조건 → 좌표 생성)과 구현 코드(`coordinate_transform_worm_to_wheel()` + `generate_worm_wheel_envelope()`)를 제공한다.

**문제**: 위 §2 코드 품질에서도 지적했듯, `generate_worm_wheel_envelope()`는 포락 조건 방정식(식 29, $\mathbf{n}_1 \cdot \mathbf{v}^{(12)} = 0$)을 **풀지 않는다**. 대신 모든 $\phi_1$에서 변환된 점을 단순 합산한다. 이는 "점 구름에서 외곽을 추출"해야 치면이 되는 근사 방법이며, 정확한 포락 생성이 아니다.

진정한 포락 생성은:
1. 각 $(u, \theta)$ 쌍에서 식 (29)를 $\phi_1$에 대해 풀어 접촉점을 찾고
2. 해당 접촉점만 수집하여 치면 메시를 구성

해야 한다. 현재 구현은 점 밀도를 높이면 근사적으로 접근하지만, 정밀 시뮬레이션에는 부족하다.

#### 4.3 자기잠금 조건 — 정확

식 (37) $\gamma < \varphi' = \arctan(\mu/\cos\alpha_n)$은 표준 교과서 수식과 일치한다. 수치 투명성 주석(마찰 계수 의존성, 실패 조건)이 적절하다. `is_self_locking()` 코드가 이를 정확히 구현하고 있다.

**단, 주의**: 자기잠금은 정적 마찰 계수 기준이며, 동적 환경(진동, 충격)에서는 동적 마찰로 교체해야 한다. 이 구분이 문서에 명시되어 있지 않다(§4.4 주석에서 "저점도 오일, 고온, 진동"을 언급하지만 "정적 vs 동적 마찰" 구분은 없다).

### 5. 새로운 누락/문제

#### 5.1 R5 웜 휠 포락의 진정한 접촉선 알고리즘 부재

위 §4.2에서 상세 지적. 포락 조건 방정식(식 29)을 수치적으로 풀어 접촉선을 추출하는 알고리즘이 필요하다. 현재의 점 구름 방식은 시각화 목적의 1차 근사로는 사용 가능하지만, 시뮬레이션 앱의 정밀 치면 생성에는 부족하다.

#### 5.2 R5 ZI 웜 코드 미구현

수식(식 21)은 있으나 `generate_zi_worm_surface()` 같은 코드가 없다. ZA 코드를 참고하면 구현 난이도는 낮지만, "복사-붙여넣기 즉시 실행 가능" 기준에는 미달.

#### 5.3 R4 `envelope_generate()`의 `transform_func` 미사용

`envelope_generate()` 함수 시그니처에 `transform_func` 인자가 있지만, 함수 본체에서 사용되지 않는다. 대신 `tool_profile_func`이 변환까지 포함하도록 설계되어 있어, `rack_cutter_envelope_demo()`에서 `transform_func=None`으로 호출한다. 인터페이스 설계가 혼란스럽다. `transform_func`을 삭제하거나 실제로 사용하도록 수정해야 한다.

#### 5.4 R4 §D.3 이끝 간섭 검사의 한계

`check_tip_interference()` 함수에서 `limit_for_gear1`과 `limit_for_gear2`가 동일한 값 `a * sin(alpha_b)`로 계산된다. 이는 외접 기어에서는 정확하지만, **내접 기어에서는 간섭 조건이 다르다**. 함수가 `gear_type` 파라미터를 받지 않으므로 내접 기어에 적용하면 오류를 낼 수 있다.

#### 5.5 R5 웜 기어 전위(profile shift) 미포함

R5 §8.1 Q2에서 "수정 웜 기어($x_{t2} \neq 0$)의 영향"을 인접 질문으로 언급하면서 "코드에 `xt2` 파라미터를 추가해야 한다"고 적었지만, 실제 코드에는 전위가 전혀 반영되지 않았다. 웜 기어에서 축간 거리 조정을 위한 전위는 실무적으로 흔하므로, 최소한 `WormGearPair`에 `xt2` 필드와 이에 따른 치수 보정 수식이 있어야 한다.

#### 5.6 R4 코드 간 의존성 미명시

R4의 코드 스니펫들은 독립적으로 작성되어 있으나, 실제로는 `safe_inv()`, `inv_involute()`, `validate_tooth_count()`, `profile_shift_limits()` 등이 다른 함수에서 참조된다(예: `compute_contact_ratio()`가 `safe_inv()`와 `inv_involute()`를 호출). 이들을 하나의 모듈로 통합할 때의 import 순서나 모듈 구조가 제안되어 있지 않다. §G에서 `GearValidator` 클래스 통합을 제안한 것은 올바르지만, 통합 코드 자체는 없다.

### 6. 종합 판정

**R4(수치 구현 보충)**: Critic이 지적한 §4.1~§4.5, §6.2의 모든 항목을 체계적으로 해소했다. 코드 품질이 높고, 엣지 케이스 처리가 철저하며, 검증 알고리즘까지 포함한 점이 우수하다. `envelope_generate()`의 `transform_func` 미사용(§5.3)과 내접 기어 이끝 간섭 미지원(§5.4)은 minor한 결함이다.

**R5(웜 기어 수학)**: Critic §6.3의 "웜 기어 완전 누락"을 해소했다. 기본 파라미터, 4가지 웜 유형 수식, 효율/자기잠금/미끄러짐 계산, 설계 표준을 포괄적으로 다루었다. 그러나 **웜 휠 포락 생성의 핵심 알고리즘(포락 조건 방정식 풀이)이 불완전**하고(§5.1), ZI/ZK 코드가 없으며(§5.2), 전위가 미반영(§5.5)이다.

**결론**:
- R4: **핵심 결함 없음. Synthesis에 그대로 반영 가능.**
- R5: **구현 가능 레퍼런스로서 70% 달성.** ZA 웜 + 기본 파라미터 계산은 즉시 사용 가능하나, 웜 휠의 정밀 포락 생성과 ZI/ZK 구현은 후속 작업이 필요하다.

**Repair Pass 트리거 여부: 불필요.** R5의 웜 휠 포락 불완전성은 synthesis에서 "점 구름 근사이며, 정밀 포락은 후속 구현 필요"로 명시하면 충분하다. 핵심 결론(기어 설계 앱의 수학 레퍼런스 완성도)을 뒤집는 수준의 결함은 아니다.
