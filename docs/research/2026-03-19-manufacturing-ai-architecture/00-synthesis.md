## 1. 핵심 결론 요약

1. **설계 지능 레이어**: Generative Design/TO(TRL 7-9)와 규칙 기반 DRC(TRL 8-9)는 양산 수준. LLM 기반 CAD 생성(TRL 3-4)은 단순 부품의 초안 생성기 역할만 가능하며, 산업 부품으로의 확장에는 데이터셋과 신뢰성 격차가 크다.
2. **공정 지능 레이어**: CAM Assist(TRL 8-9)와 공구 추천 시스템(TRL 8-9)은 양산. AI CAPP(TRL 2-4)와 G-code 자동 생성(TRL 2-3)은 연구 단계. 현실적으로 "AI 보조 + 인간 검증" 워크플로우가 1-2년 내 도달점이다.
3. **실행 지능 레이어**: 인라인 머신비전 검사(TRL 8-9)와 예측 정비 기술(TRL 8-9)은 양산이나, 예측 정비 프로젝트의 60-70%가 목표 ROI를 달성하지 못한다. 조직/데이터 품질 문제가 기술보다 큰 장벽이다.
4. **통합 아키텍처**: 3레이어 + 2횡단 요소(온톨로지 허브, 데이터 파이프라인) 구조가 핵심 제안. 단, MVP에서는 JSON Schema + 파일 교환으로 시작하고, OWL/Kafka는 검증 후 점진 도입해야 한다.
5. **가장 높은 ROI 행동**: (a) 인라인 머신비전 검사 도입(6-12개월 ROI 회수), (b) 고영향 자산 대상 예측 정비 파일럿, (c) RAG 기반 규격/공구 검색 시스템 구축 — 이 세 가지가 독립적으로도 가치를 창출하며, 통합 시 시너지를 만든다.

04번 보고서의 핵심 구조를 유지하되, Critic 이슈를 반영하여 현실적으로 조정한다. 상세 구조는 [04-integration-architecture.md §4](./04-integration-architecture.md)를 참조.

### 3.2 온톨로지 허브 전환 경로 [이슈 3-1 반영]

01번과 02번 조사에 따르면 MASON 등 제조 온톨로지의 산업 양산 사례는 극소수이며, 유지보수 비용과 레거시 통합이 핵심 병목이다. 04번이 이를 핵심 횡단 요소로 채택한 것은 시맨틱 통합의 논리적 필요성에는 근거가 있지만, 초기부터 OWL/SHACL을 요구하는 것은 비현실적이다.

**전환 경로:**

| 단계 | 스키마 기술 | 저장소 | 검증 | 비용 |
|------|-----------|--------|------|------|
| Phase 0-1 (MVP) | JSON Schema | SQLite / 파일 시스템 | JSON Schema Validator | 최소 |
| Phase 2 | JSON-LD + 경량 OWL | Neo4j Community | pyshacl 기본 규칙 | 중간 |
| Phase 3 | MASON 확장 OWL + SHACL | Neo4j Enterprise | 풀 SHACL 검증 + 추론 | 높음 |

JSON Schema → OWL 전환이 정당화되는 조건: (a) 크로스-레이어 시맨틱 추론이 실제로 필요한 유스케이스 확인, (b) Phase 1에서 JSON Schema의 한계가 측정됨.

### 3.4 Phase 1 MVP 비용 범위 추정 [이슈 4-1 반영]

| 항목 | 인력/단가 | 기간 | 비용 추정 |

Phase 2-3 추가 시 Kafka 인프라($10K-$30K/년), 온톨로지 엔지니어(희소 인력, $12K-$18K/월), Edge 디바이스 등이 추가된다. 전체 3 Phase 완료 시 $300K-$600K 범위로 추정.

### 3.5 IP 보안 아키텍처 [이슈 4-2 반영]

4개 보고서 모두에서 구조적으로 누락된 보안/IP 이슈를 보완한다.

**데이터 분류 체계:**

| 등급 | 정의 | LLM 사용 정책 | 예시 |
|------|------|-------------|------|
| 공개 | 일반 기술 정보 | 클라우드 LLM (Claude API) 사용 가능 | 규격 문서, 공개 핸드북 |
| 내부 | 사내 노하우 | 클라우드 LLM 사용 가능하되, 파라미터/수치는 마스킹 | 공정 파라미터 범위, 일반 DFM 규칙 |
| 기밀 | 핵심 IP | On-premise LLM만 허용 | CAD 모델, 고객 도면, 독점 공정 조건, 불량 데이터 |

**On-premise LLM 대안:**
- ARKNESS가 3B Llama-3으로 GPT-4o급 공정 지식 QA 성능을 달성한 사례는 on-premise 배포의 실현 가능성을 보여준다 (02번 보고서 §1.2)
- 설계 지능 레이어의 코드 생성은 대형 모델(Claude 급)이 필요하나, 공정/실행 레이어의 지식 검색은 소형 모델로 충분할 수 있다
- 하이브리드 전략: 기밀 데이터 처리는 on-premise, 일반 추론은 클라우드

### 3.6 Phase 0: SME 최소 도입 경로 [이슈 4-3 반영]

중소기업(50인 미만, IT 전담 인력 0-1명)을 위한 최소 도입 경로:

| 구성 요소 | Phase 0 (SME) | Phase 1 (MVP) |
|-----------|--------------|---------------|
| 스키마 | JSON Schema | JSON Schema → JSON-LD |
| 저장소 | SQLite / CSV | SQLite → Neo4j |
| 통신 | 파일 교환 | JSON 파일 → Kafka |
| LLM | Claude API (SaaS) | Claude API + 필요 시 on-premise |
| CAD | build123d (무료) | build123d |
| 공구 DB | CoroPlus/NOVO (기존 무료 도구) | KG 기반 확장 |
| 센서 | ESP32 + MEMS (< $50/노드) | 산업용 센서 |
| 비용 | **$10K-$30K** (3개월) | $80K-$165K |

**Phase 0 범위**: 단일 부품 유형에 대해 "자연어 → CAD 초안 + 기본 공정 계획 + CoroPlus 공구 추천" 워크플로우를 구현. 인라인 검사나 예측 정비는 포함하지 않는다.

## 6. 역방향 의사결정 가이드

| 상황 (결과가 X이면) | 조치 (Y를 조정하라) |
|---------------------|---------------------|
| Phase 1 MVP에서 JSON Schema의 한계가 드러나면 (크로스-레이어 추론 불가) | → Phase 2에서 JSON-LD + 경량 OWL 도입을 앞당겨라 |
| Phase 1 MVP에서 JSON Schema로 충분하면 | → OWL 전환을 연기하고 기능 확장에 투자하라 |
| 인라인 검사 AI 도입 후 오경보율이 높으면 | → 도메인 특화 파인튜닝 또는 NVIDIA NV-DINOv2 패턴의 SSL 적용을 검토하라 |
| 예측 정비 파일럿에서 알림 피로가 발생하면 | → 알림 우선순위화 로직 추가 + 에스컬레이션 체계 재설계. 기술이 아닌 운영 프로세스 문제일 가능성 높음 |
| LLM CAD 생성 결과의 산업 부품 적용 실패율이 30%+ 이면 | → 엔지니어 초안 보조 역할로 범위를 축소하고, 기존 설계 수정(parametric variation)에 집중하라 |
| On-premise LLM 성능이 클라우드 대비 현저히 낮으면 | → 데이터 분류를 세분화하여 민감하지 않은 작업만 클라우드로 전환하라 |
| 통합 아키텍처의 구현 비용이 $200K를 초과하면 | → Phase 0(SME) 경로로 범위를 축소하고, 단일 고영향 유스케이스에 집중하라 |
| Digital Twin 도입 시 데이터 품질이 미달이면 | → DT 전에 센서 인프라 + 데이터 거버넌스부터 구축. "DT 없이 ML만으로" 접근 검토 |
| 조직 저항이 기술 장벽보다 크면 | → Human-in-the-loop을 강화하고, AI를 "제안 도구"로만 포지셔닝. 강제 전환보다 점진적 신뢰 축적 |

## 8. 후속 액션 (우선순위 순)

### 즉시 가능 (기존 자산 활용, 1-2주)

1. **build123d 기어 모델링 + JSON Schema 검증 파이프라인 구축**
   - 기존 자산: `../2026-03-18-build123d-gear-system/01-build123d-gear-modeling.md`
   - 행동: 기어 파라미터 JSON Schema 정의 → build123d 코드 생성 → STEP 출력 → 기본 규칙 검증
   - 기대: Phase 0/1의 설계 지능 코어 동작 확인

2. **RAG 기반 규격/공구 검색 프로토타입**
   - 기존 자산: `../2026-03-18-gear-system/02-manufacturing-ontology.md`, `../2026-03-18-gear-system/04-gear-manufacturing-process-chain.md`
   - 행동: AGMA/ISO 규격 문서 + 공구 카탈로그를 벡터 DB에 적재 → LLM QA 테스트
   - 기대: 공정 지능 레이어의 지식 검색 기능 검증

### 단기 (Phase 1 MVP, 3-4개월)

3. **설계-공정 루프 end-to-end 구현**
   - 자연어 → build123d 기어 STEP → 공정 계획 JSON → DFM 검증 → 재설계 루프
   - 기존 MCP 서버 설계 활용: `../2026-03-18-build123d-gear-system/02-integrated-architecture.md`
   - 성공 기준: SHACL(또는 JSON Schema) 위반 시 자동 재설계

4. **IP 보안 정책 수립**
   - 데이터 분류 체계 확정
   - On-premise LLM 평가 (Llama 3 등으로 공정 지식 QA 테스트)

### 중기 (Phase 2, 3-4개월)

5. **센서 데이터 시뮬레이터 + 이벤트 파이프라인**
   - Python 기반 CNC 센서 데이터 시뮬레이터 → Kafka → 공정 지능 이벤트 소비
   - 품질 예측 모델(기본 ML) + KG 자동 업데이트

6. **인라인 검사 AI 파일럿 평가**
   - 기어 검사 도메인에 맞는 결함 유형 정의
   - 기존 검사 데이터로 모델 성능 사전 평가

### 장기 (Phase 3, 6-12개월)

7. **Closed-Loop 피드백 시스템**
   - 실측 데이터 → 설계 규칙 자동 갱신
   - 성공 기준 재정의 [Critic 5-6 반영]: "통합 시스템 vs 미사용 시의 개선 속도 비교"로 측정

8. **Digital Twin 타당성 평가**
   - Phase 2 데이터 품질/센서 인프라 상태에 따라 DT 도입 여부 결정
   - 02번 보고서의 5대 장벽 체크리스트 통과 시에만 진행
