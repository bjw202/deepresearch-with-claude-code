# Critic Review: 팔란티어 온톨로지 리서치

> **역할**: Critic **작성일**: 2026-03-23 **검토 대상**: 01-technical-architecture.md (R1), 02-industry-cases.md (R2), 03-aip-evolution-limits.md (R3) **사용자 원래 질문**: "팔란티어에서 내세우는 온톨로지로 어떤 문제들을 해결하고 어떻게 운영하고 있는지 실제적인 사례들을 상세하게 리서치해줘. 단순히 결과가 아니라 어떤 방식으로 온톨로지를 구축하고 운영하는지가 기술적으로 그리고 원리적으로 잘 나타나야 한다."

## 체크리스트 1: 도메인 적용성

> 인용된 연구/데이터가 대상 도메인에 유효한가

### 1.1 팔란티어 공식 문서 — 유효 | 심각도: 낮음

R1은 Palantir Foundry 공식 문서를 주 출처로 사용하며, 인용이 정확하고 도메인이 직접 일치한다. Object Type, Link, Action, Function의 정의는 공식 문서 원문을 직접 인용하고 있어 도메인 적용성이 높다.

### 1.2 인접 도메인 출처의 무비판적 적용 우려 | 심각도: 중간

- **R3의 Ruohang Feng(Pigsty 창업자) 비판**: Feng은 PostgreSQL 기반 인프라 전문가이지, 팔란티어 Foundry의 실제 사용자가 아니다. 그의 "온톨로지 = 과대포장된 DB 모델링" 비판은 기술적으로 유효한 관점이지만, R3에서 이 비판의 근거 강도를 "높음"으로 평가한 것은 과도하다. Feng은 팔란티어를 직접 사용한 경험이 없으며, 그의 분석은 외부 관찰에 기반한다. **R3 자신도 "Feng 자신이 팔란티어의 가치는 온톨로지 개념이 아닌 그 밖의 모든 것에 있다고 인정한다"고 기술하면서도 비판 자체의 근거 강도를 높음으로 매긴 것은 내적 모순이다.**

- **R3의 Vladimir Kozlov(LinkedIn, 2025) 인용**: Kozlov의 기술적 배경이 명시되지 않았다. LinkedIn 게시물은 동료 검토를 거치지 않은 개인 의견이다.

### 1.3 N3C 사례의 도메인 특수성 | 심각도: 낮음

R2의 N3C(COVID-19 코호트) 사례는 팔란티어가 자체 온톨로지가 아닌 OMOP CDM이라는 **업계 표준**을 수용한 사례다. R2는 이 차이를 명시했으나, 이 사례가 "팔란티어 온톨로지 구축 방식"의 대표 사례로 적절한지에 대한 논의가 부족하다. 팔란티어가 온톨로지를 정의한 것이 아니라 기존 표준을 수용한 것이므로, "온톨로지를 어떻게 구축하는가"라는 질문에 대한 답변력이 약하다.

## 체크리스트 2: 수치 근거 검증

> 핵심 수치의 출처와 조건이 명시되었는가

### 2.1 팔란티어 자체 마케팅 수치의 과도한 의존 | 심각도: **높음**

세 보고서 모두에서 핵심 성과 수치가 **팔란티어 자체 자료**에서 나왔으며, 독립 검증이 없다:

| 수치 | 출처 | 독립 검증 | 보고서 |
| --- | --- | --- | --- |
| A350 납품 33% 가속화 | 팔란티어-에어버스 공동 PDF | 없음 | R2 |
| KYC 처리 10일→10시간 | 팔란티어 AML 브로슈어 | 없음 | R2 |
| 진양성율 40배 향상 | 팔란티어 AML 브로슈어 | 없음 | R2 |
| 비용 90% 절감 | 팔란티어 AML 브로슈어 | 없음 | R2 |
| "days vs. years" 배포 속도 | 팔란티어 마케팅 | 없음 | R3 |
| AIP Bootcamp 1,000+ 기업 | 팔란티어 발표 | 없음 | R3 |
| 상위 20개 고객 평균 $93.9M/년 | Annual Report + vonng.com | 부분적 (IR 기반) | R3 |

**문제**: R2는 각 수치에 "수치 투명성" 주석을 달아 출처 편향을 명시했고, R3도 자체 주장임을 밝혔다. 이 점은 긍정적이다. 그러나 **보고서 전체를 통틀어 팔란티어 외부에서 나온 정량적 성과 데이터가 단 하나도 없다**는 사실은 별도로 강조되어야 한다. 이는 개별 수치의 문제가 아니라 리서치 전체의 구조적 한계다.

### 2.2 Entity Resolution 임계값 미공개 | 심각도: 중간

R2가 Gotham의 Entity Resolution에서 "유사도 임계값(s_ij &gt; θ)"를 언급하면서 θ가 비공개임을 명시한 것은 적절하다. 그러나 이 비공개성의 함의 — 외부에서 시스템의 정확성을 검증할 수 없다는 점 — 이 더 강조되어야 한다. 특히 국방/이민 분야에서 오탐(false positive)은 무고한 사람의 인권 침해로 이어질 수 있다.

### 2.3 Azule Energy 20만 bpd 수치 | 심각도: 중간

R2가 BP 사례에서 "20만 bpd 생산 최적화"를 기재했으나, 이것이 "생산량 자체가 20만 bpd"인지 "20만 bpd만큼 최적화(증가)했다"는 것인지 모호하다. 출처가 "Palantir/SPE Journal"로 병기되어 있어 SPE Journal이 독립 검증 출처인지 팔란티어 기고문인지 불분명하다.

## 체크리스트 3: 서브에이전트 간 상충점

> Researcher 산출물 간 모순이 있는가

### 3.1 온톨로지의 기술적 독창성 평가 불일치 | 심각도: **높음** — 핵심 결론을 바꿀 수 있는 결함

**R1의 입장**: 팔란티어 온톨로지는 OWL/RDF와 "근본적으로 다른 설계 철학"을 가지며, 시맨틱 웹의 개념을 "실용적으로 재해석"한 것이다. 데이터+로직+액션+보안의 4중 통합이 차별점이며, "의사결정을 모델링하는 시스템"이라는 고유한 정체성을 가진다.

**R3의 입장**: 팔란티어 온톨로지의 4개 프리미티브는 DB의 Table/Column/FK/Stored Procedure와 "실질적으로 동형"이다. "같은 것에 다른 이름을 붙여 3자리 수 배의 가격을 받는다"(Feng). R3는 이 비판의 근거 강도를 "높음"으로 평가.

**R2의 입장**: 사례 중심 서술이므로 이 논쟁에 직접 참여하지 않으나, 모든 사례에서 온톨로지 도입 전/후 비교를 제시하며 암묵적으로 온톨로지의 실질적 가치를 긍정.

**상충점**: R1은 팔란티어 온톨로지가 개념적으로 고유한 기여를 한다고 보고, R3는 그것이 과대포장된 기존 개념이라고 본다. 이 상충은 리서치의 핵심 질문 — "온톨로지를 어떻게 구축하고 운영하는가의 기술적/원리적 실체" — 에 직접 영향을 미친다. **Synthesis에서 이 상충을 해결하지 않으면, 독자는 팔란티어 온톨로지가 혁신인지 마케팅인지 판단할 수 없다.**

**Critic 판단**: 양쪽 모두 부분적으로 옳다. 팔란티어 온톨로지의 개별 구성요소(Object Type, Link 등)는 DB 모델링과 동형이라는 R3의 지적이 기술적으로 유효하다. 그러나 R1이 기술한 4중 통합(데이터+로직+액션+보안)을 **단일 플랫폼에서 비기술자가 사용할 수 있는 형태로 제공**한다는 점은 기존 DB 설계와 다른 제품 레벨의 차별점이다. 혁신은 개념이 아니라 통합과 접근성에 있다. Synthesis에서 이 구분을 명확히 해야 한다.

### 3.2 벤더 락인 심각도 평가 차이 | 심각도: 중간

**R3**: 벤더 락인 근거 강도 "높음". HASH.ai 분석을 주 근거로 사용. "완전 폐쇄 소스"로 표현. **R1**: OSDK를 통한 외부 접근, JSON 내보내기 지원을 기술. 온톨로지 정의가 JSON으로 다운로드 가능함을 명시.

R3가 "완전 폐쇄 소스"라고 단정한 반면, R1은 OSDK(오픈소스 GitHub 저장소 존재), JSON 내보내기, OpenAPI Spec 내보내기를 구체적으로 기술했다. R3도 이에 대한 반증을 언급하긴 했으나 ("OSDK로 일부 반박 가능"), 경쟁사 비교표에서는 "완전 폐쇄 소스"로 일괄 표시했다. 정확한 표현은 \*\*"코어 플랫폼은 폐쇄 소스이나, SDK와 API는 공개"\*\*이다.

### 3.3 AIP 에이전트의 온톨로지 활용 범위 | 심각도: 낮음

R1은 "AI 에이전트와 온톨로지" 관계를 관점 확장에서 언급하고, R3는 AIP Agent Studio의 멀티스텝 에이전트를 상세히 기술했다. 직접적 모순은 없으나, R1이 제기한 "에이전트가 Action을 실행할 수 있는가?"라는 질문에 R3가 이미 "AIP Agent Studio가 재고 확인 → 공급업체 평가 → 주문 생성 같은 연속 작업을 지원한다"고 답한 셈이다. 이 연결이 명시적으로 이루어지지 않았다.

## 체크리스트 4: 누락 관점

> 비용, 실행 가능성, 실패 모드가 다뤄졌는가

### 4.1 구축 비용의 구체성 부족 | 심각도: **높음**

사용자의 질문은 "어떻게 구축하고 운영하는지"를 묻고 있다. 세 보고서 모두 온톨로지 구축의 **실제 프로세스**에 대한 상세도가 불균일하다:

- **R1**: Object Type 생성의 두 가지 경로(Ontology Manager / Pipeline Builder)를 공식 문서 기반으로 상세히 기술. 가장 구체적.
- **R2**: 사례별로 어떤 데이터를 통합했는지 기술하지만, 구축 과정(얼마나 걸렸는지, 몇 명이 투입됐는지, 어떤 순서로 Object Type을 정의했는지)은 거의 없음.
- **R3**: 비용($93.9M, $4.7M)과 FDE 모델을 언급하지만, 구체적인 구축 타임라인이나 인력 구성은 기술하지 않음.

**누락**: 실제 고객이 온톨로지를 처음부터 구축하는 일반적인 프로젝트 타임라인, 팀 구성(도메인 전문가, 데이터 엔지니어, FDE의 비율), 단계별 산출물이 없다. 이것이 사용자 질문의 핵심 — "어떤 방식으로 온톨로지를 구축하고 운영하는지" — 에 대한 가장 큰 공백이다.

### 4.2 실패 사례 부족 | 심각도: 중간

R2는 ICE 사례에서 윤리적 문제를 다루었고, R3는 벤더 락인과 과대포장 비판을 다루었다. 그러나 **온톨로지 구축 자체의 실패 사례** — 온톨로지를 도입했으나 사용되지 않거나, 구축 비용을 초과하거나, 결국 폐기된 사례 — 가 없다.

R3가 도널드 파머의 1990년대 자동차 금융회사 사례("불완전한 온톨로지는 틀린 것이다")를 인용한 것은 유일한 실패 참조이나, 이는 팔란티어 온톨로지가 아닌 일반 온톨로지 프로젝트 사례다.

### 4.3 경쟁 솔루션 대비 마이그레이션 경로 | 심각도: 낮음

R3가 Databricks/Snowflake 비교표를 제공했으나, 실제로 팔란티어에서 다른 플랫폼으로 전환한 사례나, 반대로 다른 플랫폼에서 팔란티어로 이전한 사례가 없다. 비교가 정적인 기능 매트릭스에 머물러 있다.

### 4.4 온톨로지 거버넌스 운영 실무 | 심각도: 중간

R1이 스키마 변경 관리, DRY 원칙, 데이터 신선도를 공식 문서 기반으로 기술했고, R3도 유지보수 리스크를 다뤘다. 그러나 실제 조직에서 온톨로지 거버넌스가 어떻게 운영되는지 — 변경 승인 프로세스, 온톨로지 스튜어드(steward)의 역할, 팀 간 Object Type 충돌 해결 절차 — 에 대한 실무 정보가 부족하다.

## 체크리스트 5: 확신도 교정

> 근거 대비 과도한 확신이 있는가

### 5.1 R2의 성과 수치에 대한 암묵적 확신 | 심각도: 중간

R2는 각 사례의 "도입 전/후" 비교표에서 팔란티어 자체 수치를 사용하면서 "수치 투명성" 주석을 달았다. 이 주석은 적절하지만, 비교표 자체의 시각적 구조가 "도입 전 = 나쁨, 도입 후 = 좋음"이라는 이분법적 인상을 준다. 실제로는 팔란티어 도입과 무관한 요인(추가 인력, 프로세스 개선, 시간 경과에 따른 자연 성숙)이 기여했을 가능성이 있으며, 이 가능성이 표 구조에서는 보이지 않는다.

### 5.2 R3의 "과대포장" 비판에 대한 과도한 확신 | 심각도: 중간

R3가 Feng의 "온톨로지 = DB 모델링" 비판 근거 강도를 "높음"으로 매긴 것은 과도하다 (3.1에서 상세 기술). Feng의 분석은 기술적 구조 비교로는 유효하지만, 제품 가치(통합, UI, 보안, 도메인 전문가 서비스)를 포함한 전체 평가로는 불완전하다. "중간"이 더 적절하다.

### 5.3 R1의 "의사결정을 모델링하는 시스템" 프레이밍 | 심각도: 낮음

R1은 팔란티어 공식 문서의 "데이터가 아니라 의사결정을 표현하는 시스템"이라는 정의를 수용하고 이를 "기존 데이터 카탈로그와 구별하는 첫 번째 원칙"이라고 서술했다. 이는 팔란티어의 자기 정의를 그대로 채택한 것이다. 비판적 시각에서 보면, 모든 비즈니스 데이터 모델은 궁극적으로 의사결정을 지원하기 위해 존재하므로 이 구분이 실질적인지 의문이 있다. R1은 반증(Iannella의 "just structured YAML" 의견)을 기술했으나, 본문 흐름에서 팔란티어의 자기 서술에 기울어져 있다.

## 체크리스트 6: 문제 정의 검토

> 사용자의 원래 질문이 과도하게 좁거나, 중요한 리스크를 가리거나, 솔루션에 과도하게 고정되어 있지 않은가

### 6.1 사용자 질문의 팔란티어 고정 | 심각도: 중간

사용자 질문은 "팔란티어에서 내세우는 온톨로지"에 초점이 맞춰져 있다. R3가 문제 재정의에서 "기술적 독창성이 아니라 ROI가 핵심"이라고 제안했고, R2는 "어떤 결정 권한이 어디로 이동하는가"라는 권력 구조 분석을 제안했다. 두 제안 모두 유효하지만, 사용자의 원래 의도 — 팔란티어 온톨로지의 기술적/원리적 작동 방식을 이해하고 싶다 — 와는 다소 괴리가 있다.

사용자 질문에서 빠져 있지만 중요한 관점:

- **대안 검토**: 팔란티어 없이 유사한 온톨로지 기반 운영 레이어를 구축할 수 있는가? (오픈소스 조합: Apache Atlas + Airflow + dbt + 커스텀 액션 레이어 등)
- **규모별 적합성**: 어떤 규모/복잡도의 조직에서 팔란티어 온톨로지가 의미 있는가? 스타트업/중소기업에는 과잉인가?

### 6.2 "기술적으로 그리고 원리적으로" 요구의 충족도 | 심각도: 중간

사용자는 "어떤 방식으로 온톨로지를 구축하고 운영하는지가 기술적으로 그리고 원리적으로 잘 나타나야 한다"고 요청했다.

- **원리적 측면**: R1이 설계 철학(OWL 대비 차이, 4중 통합, 디지털 트윈)을 잘 다룸. 충족.
- **기술적 구축 측면**: R1이 Object Type 생성 방법, 데이터 흐름, 백엔드 마이크로서비스를 상세히 기술. 충족.
- **운영 측면**: R1이 유지보수 가이드라인을 기술했으나, 실제 운영 팀의 일상 워크플로우(일일 모니터링, 온톨로지 변경 요청 처리, 장애 대응)는 부족. **부분 충족**.
- **사례의 구축 과정**: R2가 산업별로 "어떤 Object Type을 만들었는가"는 상세히 기술했으나, "어떤 순서로, 얼마나 걸려서, 누가 만들었는가"는 미흡. **미충족**.

### 핵심 검증: 역설계 가능성 평가

#### Object Type 식별과 정의 | 평가: **충족** | 심각도: 낮음

세 보고서 모두 Object Type 설계에 대해 충분한 수준의 가이드를 제공한다.

- DR1은 Airbus 도메인에서 Aircraft, Flight, Component, SensorReading, MaintenanceAlert, WorkOrder, Airline 7개 Object Type을 하나씩 분해하면서 **"이것이 왜 필요한가"를 일관되게 설명**한다. 각 Object에 대해 속성 표, 출처 시스템, "이 속성이 없으면 어떻게 되는가"까지 기술하여 설계 의도가 명확하다.
- DR2은 AML 도메인에서 Customer, Account, Transaction, Alert, Case, Organization 6개를 동일한 패턴으로 분해한다. 특히 Customer의 25개 위험 지표를 속성 단위로 설명한 것은 실무 설계에 직접 활용 가능한 수준이다.
- DR3은 도메인 독립적인 5개 설계 원칙(허브 패턴, 이벤트 분리, PK 전략, 네이밍, Property vs Object 분리 기준)을 체계적으로 정리했다. 특히 **판단 기준 표**(별도 Object로 분리해야 하는 4가지 질문)는 실전에서 바로 사용할 수 있는 도구다.

**단, 한 가지 부족한 점**: Object Type 식별의 **최초 단계** — 도메인 전문가와의 워크숍에서 어떤 질문을 던져서 후보 Object를 추출하는가 — 에 대한 프로세스가 DR3의 "허브 Object 식별 질문"(한 줄) 외에는 구체적이지 않다. DR1과 DR2는 이미 Object가 결정된 상태에서 시작한다. \*\*"아직 아무것도 없는 상태에서 첫 Object를 어떻게 찾는가"\*\*에 대한 발견적 방법론이 보강되면 완전해진다.

#### Link 설계 (1:N vs M:N, 방향성) | 평가: **충족** | 심각도: 낮음

- DR1은 모든 Object Type에 Link 표(대상, 카디널리티, 설계 이유)를 일관되게 포함했다. WorkOrder → Component가 N:N인 이유("하나의 작업 지시서가 여러 부품을 다룰 수 있음")도 명시되어 있다.
- DR2은 Link 표에 **"없으면 놓치는 것"** 컬럼을 추가하여 Link의 존재 이유를 인과적으로 설명했다. 특히 Customer → Customer (N:M, 공유 주소/IP 기반)의 자기 참조 Link가 AML 네트워크 분석의 핵심임을 잘 보여준다.
- DR3은 1:N vs M:N 선택 기준을 표로 정리하고, **조인 Object가 필요한 4가지 조건**(관계에 속성 존재, 이력 추적, Action 필요, 관계 자체 속성)을 명확히 했다. Link 방향성이 쿼리 패턴에 미치는 영향도 예시와 함께 설명했다.

이 세 보고서를 종합하면 **Link 설계에 대해 충분히 역설계 가능하다**고 판단한다.

#### Action과 Writeback의 전체 체인 | 평가: **우수** | 심각도: 없음

이 부분이 3개 보고서의 **가장 강력한 기여**다.

- DR1의 "센서 교체 완료" 시나리오는 Action의 완전한 체인을 단계별로 분해한다: 파라미터 → Submission Criteria → Ontology Rules (5개) → Side Effect Rules (2개) → Writeback → 다운스트림 파이프라인 트리거. 특히 구 부품 장착 해제와 신 부품 장착을 하나의 트랜잭션에서 처리하는 규칙까지 보여줬다.
- DR2의 "의심 거래 확인" / "오탐 해제" 두 시나리오는 Action의 Side Effect가 ML 피드백 루프로 연결되는 과정을 코드 수준으로 추적한다. Alert → Customer risk_score 재계산 → Case 생성 → SAR 자동 생성의 연쇄를 보여준 것은 역설계에 직접 활용 가능하다.
- DR3은 Action의 3요소(파라미터 → 규칙 → Side Effect)를 원리 수준에서 정리하고, Writeback Webhook vs Side Effect Webhook의 차이를 "외부 실패 시 온톨로지 변경 취소 여부"로 명확히 구분했다. Writeback의 7단계 완전한 체인(버튼 클릭 → Submission Criteria → Writeback Webhook → Ontology Rules → Writeback Dataset → Object Storage → Side Effect/Notification)은 이 리서치 전체에서 가장 구체적이고 유용한 산출물이다.

**세 보고서 통합 시 Action/Writeback에 대한 역설계 가능성은 높음**으로 평가한다.

#### Security 모델 | 평가: **충족** | 심각도: 낮음

- DR1은 Skywise의 다중 테넌트 보안을 Organization 격리 → 행 수준 보안 → 열 수준 보안 순서로 3계층 구조를 구체적으로 보여줬다. 특히 팔란티어 공식 문서의 Sky Industries 예시를 Airbus/항공사에 정확히 매핑했다.
- DR3은 보안 모델을 일반화하여 Object Security Policy(행), Property Security Policy(열), Action-only Edit를 설명하고, **Link를 통한 보안 전파가 일어나지 않는다**는 핵심 사실을 명시했다. AIP에서 LLM이 현재 사용자 권한 범위만 따르는 메커니즘도 포함했다.
- DR2는 AML에서 감사 로그(Audit Log)의 불변성과 규제기관 제출 가능성을 다뤘다.

**부족한 점**: 보안 정책을 **처음부터 설계하는 실무 순서**가 DR3의 체크리스트 단계 7에 3줄로만 기술되어 있다. "어떤 조건에서 행 수준 보안이 필요한가", "열 수준 보안이 과도한 경우는 어떤 신호인가" 같은 판단 기준이 더 있으면 좋겠지만, 이는 미미한 수준이다.

### 표준 6개 체크리스트

#### 1. 도메인 적용성 | 심각도: 낮음

세 보고서 모두 출처의 도메인 적용성이 높다.

- DR1: 팔란티어 공식 문서의 항공 예시(Sky Industries, Aircraft Object)를 직접 적용. Airbus 공식 뉴스룸과 Flying Magazine을 교차 검증. **단, DR1이 "추론 포함"으로 명시한 부분** — 실제 Skywise 내부 스키마는 비공개이므로 역설계임 — 이 정직하고 적절하다.
- DR2: 팔란티어 AML 공식 문서(PDF 3종)를 주 출처로 사용. 오탐률 90\~95%는 Flagright, Unit21, Retail Banker International 등 **다수의 독립 출처가 일치**하여 도메인 적용성 높음.
- DR3: 팔란티어 공식 문서 22개를 주 출처로 사용. 커뮤니티 포럼(community.palantir.com) 인용도 적절. Fourth Age 블로그 인용은 비공식 출처이나 실무 가이드로서 보완적 가치가 있다.

#### 2. 수치 근거 검증 | 심각도: **중간**

- DR1의 "A350 납품 33% 가속화"와 "12,000대 항공기, 50,000명 사용자"는 출처를 명시했으나, 전자는 팔란티어-에어버스 공동 자료(독립 검증 없음), 후자는 Airbus 뉴스룸 + Flying Magazine(독립 확인됨)이다. DR1이 출처 평가 섹션에서 이 차이를 구분한 것은 적절하다.
- DR2의 "40배 진양성 개선", "비용 90% 절감"은 **여전히 팔란티어 자체 마케팅 수치**이며, 1차 Critic에서 지적한 문제가 해소되지 않았다. DR2가 수치 투명성 주석("특정 유럽 대형 은행 1곳의 배포 사례")을 달아 조건을 명시한 것은 1차 대비 개선이지만, 독립 검증의 부재라는 구조적 한계는 여전하다.
- DR3의 스트리밍 지연 시간("총 6-7초")은 팔란티어 공식 문서 기반이며, "이 수치가 틀릴 수 있는 조건"까지 명시하여 투명성이 높다.

**종합**: DR2의 성과 수치에 대해서는 Synthesis에서 "팔란티어 자체 주장이며 독립 검증 미확인"을 반복 강조해야 한다. 이는 1차 Critic의 지적(체크리스트 2.1)과 일관된다.

#### 3. 보고서 간 상충점 | 심각도: **낮음** (1차 대비 개선)

3개 보고서 간 **직접적 모순이 발견되지 않는다**. 오히려 보완 관계가 강하다:

- DR1(항공)과 DR2(금융)는 같은 온톨로지 원리를 다른 도메인에 적용한 사례로, 패턴의 일반성을 증명한다. DR1의 "이벤트 → 경보 → 조사 → 해결" 4단계 체인과 DR2의 "Transaction → Alert → Case → SAR" 체인이 구조적으로 동형이라는 점은 DR3의 설계 패턴을 실증적으로 뒷받침한다.
- DR3은 DR1, DR2에서 추출한 패턴을 도메인 독립적 원칙으로 일반화했다. 이 과정에서 왜곡이나 과대 일반화는 발견되지 않았다.

**유일한 미세 불일치**: DR1은 Skywise의 허브 Object를 `Aircraft`로 설정했으나, DR3의 도메인별 예시 표에서 항공사 허브를 `Flight`로 기재했다. 이는 "항공사 운영"(Flight 중심)과 "항공 MRO/제조"(Aircraft 중심) 관점 차이이며, 모순이 아니라 **관점 차이**다. 다만 Synthesis에서 "허브 선택은 비즈니스 질문에 따라 달라진다"는 점을 명시해야 한다.

#### 4. 누락 관점 | 심각도: **중간**

**(a) 온톨로지 구축 실패 시나리오** | 심각도: 중간

1차 Critic에서 지적한 "온톨로지 구축 자체의 실패 사례 부족"이 보완 리서치에서도 해소되지 않았다. DR3이 "이 설계 원칙이 작동하지 않는 조건" 3가지(단순 도메인, 고빈도 시계열, 레거시 통합)를 기술한 것은 진전이지만, **실제로 온톨로지 설계를 잘못하여 프로젝트가 실패한 구체적 사례**는 여전히 없다. DR3의 "한 제조업체에서 복합 키를 사용하다 리팩토링한" 사례가 유일한 실패 참조다.

**(b) 온톨로지 진화(Evolution) 관리** | 심각도: 중간

DR3이 "버전명 금지"와 "Object Type은 진화 대상"이라고 기술했으나, **실제로 운영 중인 온톨로지가 변경될 때 무엇이 일어나는가**에 대한 설명이 불충분하다. 구체적으로:

- Property를 추가할 때: 기존 Object 인스턴스에 null이 들어가는가?
- Property를 삭제할 때: 기존 데이터는 어떻게 되는가?
- Object Type을 삭제할 때: 연결된 Link는 자동으로 정리되는가?
- 대규모 스키마 변경의 롤백 절차는 무엇인가?

이 질문들은 "새 도메인에서 온톨로지를 설계할 수 있는가"보다 "운영할 수 있는가"에 해당하며, 사용자의 원래 질문("어떻게 운영하는지")에 직접 관련된다.

**(c) 구축 비용과 타임라인** | 심각도: 중간

1차 Critic의 4.1에서 지적한 "구축 비용의 구체성 부족"이 여전히 해소되지 않았다. DR1은 구축 과정을 기술적으로 상세히 다뤘으나, "Skywise 초기 구축에 몇 명이 몇 개월 걸렸는가"는 없다. DR2도 "프랑스 대형 은행에서 26개 시나리오를 2주 만에 작성"이라는 단편적 수치만 있다. 이는 팔란티어의 FDE(Forward Deployed Engineer) 모델과 함께 이해해야 하는 정보인데, 보완 리서치에서도 FDE 투입 규모/기간에 대한 데이터는 확보되지 않았다.

#### 5. 확신도 교정 | 심각도: 낮음

보완 리서치 3개 보고서는 1차 리서치 대비 **확신도 관리가 크게 개선**되었다.

- DR1: "추론 포함 — 실제 Skywise 내부 스키마는 비공개이므로 역설계"를 명시적으로 밝힘. 반증 탐색에서 "Airbus가 팔란티어 의존도를 낮추려 한다"는 업계 동향까지 언급.
- DR2: 핵심 수치마다 "이 수치가 틀릴 수 있는 조건"을 기술. 피드백 루프의 악용 가능성(FinCEN Files)을 "반증 미발견이 아니라 실제 확인된 사례"로 정확히 분류.
- DR3: "반증 미발견" 항목을 명시하고, 한계 조건(단순 도메인, 고빈도 데이터, 레거시 통합)을 솔직하게 기술.

**과도한 확신은 발견되지 않음.**

#### 6. 쉬운 언어 준수 | 평가: **우수** | 심각도: 없음

세 보고서 모두 **1차 리서치 대비 현저한 개선**을 보여준다. 사용자 피드백("쉬운 언어")이 충실히 반영되었다.

- DR1: Object Type → "엑셀 시트의 컬럼 템플릿", Link → "VLOOKUP보다 강력한 양방향 연결", Pipeline Builder → "엑셀 파워쿼리와 비슷", Writeback → "구글 시트에서 수정하면 원본이 업데이트되는 것" 등 **모든 기술 개념에 일상적 비유가 선행**한다.
- DR2: 온톨로지 → "구글 지식 패널", 데이터 사일로 → "병원 진료과 간 정보 미공유", 쉘 컴퍼니 → "여러 국가에 우편함을 만드는 것", SQL 한계 → "층수를 미리 알아야 하는 사다리 타기" 등 비전문가가 따라올 수 있는 설명이다.
- DR3: 온톨로지 전체 → "도시를 건설하는 것", Interface → "220V 표준 플러그", Action → "결재 시스템", Derived Property → "엑셀 수식 셀" 등 큰 그림 비유로 시작하여 세부로 들어가는 구조가 일관된다.

### 추가 검증

#### 인과관계 설명 충분성 | 평가: **우수** | 심각도: 없음

"A가 바뀌면 B가 바뀐다"의 인과 체인이 3개 보고서 모두에서 명확히 기술되어 있다.

- DR1: Aircraft.totalFlightHours 증가 → Component.remainingLifeHours 재계산 → MaintenanceAlert 생성 → WorkOrder 생성. 이 4단계 연쇄가 자동으로 발생함을 명시했다.
- DR2: Transaction 생성 → ML Function 실행(위험 점수 계산) → 시나리오 엔진 병렬 실행 → Alert 생성 → 분석관 할당 → 판정 → 피드백 루프. **Transaction 하나의 완전한 생애주기**를 0단계\~6단계로 추적한 것은 이 리서치 전체에서 가장 인과적으로 완전한 서술이다.
- DR3: Writeback의 7단계 체인에서 각 단계의 실패 시 행동까지 기술 (Writeback Webhook 실패 → 롤백, Side Effect 실패 → 변경 유지).

#### 구체적 시나리오 충분성 | 평가: **우수** | 심각도: 없음

사용자 피드백에서 요청한 "센서 교체 완료 버튼 클릭 후..." 수준의 구체적 시나리오가 충분히 포함되어 있다.

- DR1: "엔지니어 김민준이 HL8082 항공기의 오일 압력 센서를 교체하고 완료 버튼을 누른다" → CompleteWorkOrder Action의 파라미터, 5개 Ontology Rule, 2개 Side Effect Rule, Writeback 과정까지 완전 추적
- DR1: "MSN523의 엔진 오일 온도 센서 데이터"가 비행기 안(ACMU) → ACARS → Raw Dataset → Transform → Ontology Output → Phonograph 인덱싱까지 도달하는 6단계 추적
- DR2: "고객 A가 고객 B에게 $9,500 국제 송금" → 코어뱅킹 원본 → 파이프라인 정제 → Transaction Object 생성 → ML 위험 점수 계산 → 시나리오 엔진 검사 → Alert 생성 → 분석관 할당의 완전한 추적
- DR2: "오탐 해제 버튼" → FP 레이블 저장 → 학습 데이터셋 추가 → ML 재학습 트리거의 피드백 루프까지

#### "왜 이렇게 설계했는가" 설명 | 평가: **충족** | 심각도: 낮음

- DR1: 모든 Object Type에 "이것이 왜 필요한가" 섹션을 포함. "Component Object Type이 없으면 잔여 수명 추적 불가" 같은 부재 시 영향을 일관되게 설명한다.
- DR2: 온톨로지 설계 역공학 섹션(7.1)에서 "탐지하고 싶은 패턴을 먼저 정의한다"는 **목적 주도 설계 순서**를 제시했다. 이것이 "왜"에 대한 가장 강력한 답변이다.
- DR3: Property vs Object 분리 기준을 4가지 질문으로, 1:N vs M:N vs 조인 Object를 4가지 조건으로 정리했다.

**부족한 점**: DR3의 5개 원칙 중 원칙 4(네이밍 규칙)는 "왜" 설명이 있지만("EmployeeV2가 존재하면 혼란"), 원칙 3(PK 전략)의 "왜 단일 비즈니스 ID여야 하는가"는 설명이 짧다. "복합 키를 쓰면 모든 참조가 복잡해진다"는 이유만으로는 초심자가 왜 UUID가 차선인지 완전히 이해하기 어렵다. 실무적으로는 UUID가 더 안전한 경우도 있기 때문이다.

### 종합 판정

#### 핵심 결론을 바꿀 수 있는 결함

**없음.** 보완 리서치 3개 보고서는 1차 Critic에서 지적된 핵심 결함(기술적 독창성 평가 상충, 독립 검증 부재)에 직접 대응하지는 않았으나(이는 Synthesis의 역할), **사용자 피드백에서 요구한 "역설계 가능성"과 "쉬운 언어"를 충실히 달성**했다. 보고서 간 상충이 없고, 확신도 관리가 양호하며, 인과 체인과 구체적 시나리오가 풍부하다.

#### Repair Pass 필요 여부

**불필요.** 누락 관점(실패 사례, 스키마 진화, 구축 비용)은 Synthesis에서 "이 리서치에서 답하지 못한 질문"으로 명시하면 충분하다. 추가 조사가 필수적인 수준의 결함은 없다.

#### Synthesis 권고사항 (2차)

1. DR1과 DR2의 도메인별 인과 체인이 **구조적으로 동형**("이벤트 → 경보 → 조사 → 해결")임을 명시적으로 연결하여, 이 패턴이 도메인 독립적 설계 원리임을 강조할 것
2. DR3의 "허브 Object 식별" 질문("우리 조직이 매일 아침 가장 먼저 확인하는 것은?")과 DR1의 Aircraft/DR2의 Transaction 선택을 연결하여, **허브 선택이 비즈니스 질문에 따라 달라진다**는 점을 명시할 것 (DR3 표에서 항공사=Flight vs DR1에서 Skywise=Aircraft의 미세 불일치 해소)
3. 1차 Critic의 핵심 지적(독립 검증 부재, 기술적 독창성 상충)은 보완 리서치에서 해소 대상이 아니었으나, Synthesis에서 **반드시 다뤄야 한다**
4. "이 리서치에서 답하지 못한 질문"으로 아래를 명시할 것:
   - 온톨로지 구축 프로젝트의 일반적 타임라인과 팀 구성
   - 운영 중 스키마 변경(Property 추가/삭제, Object Type 폐기)의 실무 절차
   - 온톨로지 설계 실패 사례와 복구 과정
