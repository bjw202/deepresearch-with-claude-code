# 온톨로지 + LLM 멀티에이전트 기반 제조 불량 분석-레시피 제어 시스템: 종합 보고서

> **읽는 순서**: 이 문서를 먼저 읽고, 관심 영역에 따라 상세 보고서(01~03)로 진입한다.

---

## 한 문장 결론

**기술적으로 가능하지만, "온톨로지+LLM으로 레시피를 자동 생성"하는 구조가 아니라 "LLM이 판단하고, 전통 ML이 수치를 계산하며, 온톨로지가 안전을 보장하는" 3계층 하이브리드 구조가 현실적이다. 기존 APC/R2R을 대체하는 것이 아니라 그 위에 지능형 보조 계층으로 시작해야 한다.**

---

## 1. 핵심 질문과 답변

### 사용자의 원래 질문

> 온톨로지 기반 지식그래프 + LLM 멀티에이전트로 "불량 감지 → 원인 분석 → 레시피 조정 → 사람 승인 → 설비 전달" 폐쇄 루프가 가능한가?

### 3개 보고서 + Critic이 수렴한 답변

**조건부 가능. 단, 3가지 핵심 조건이 필요하다:**

1. **LLM이 혼자 수치를 결정하면 안 된다** — LLM은 "어떤 파라미터를 조정해야 하는지" 판단하고, 구체적 수치는 전통 ML/물리 모델이 계산해야 한다. LLM의 수치적 추론은 구조적으로 취약하다 (ACL 2025, arXiv:2509.06332).

2. **기존 시스템을 대체하지 말고 보완해야 한다** — 이미 SPC/FDC/APC/R2R이 있는 현장에서, 이것들을 걷어내고 새 시스템을 넣는 것은 위험하다. 기존 인프라 위에 "지능형 보조 계층"으로 올리는 것이 현실적이다.

3. **먼저 "이 시스템이 정말 필요한가"를 확인해야 한다** — 안정적 공정에서 R2R이 95-98% 정확도를 유지하고 있다면, AI를 추가해도 개선 폭이 미미하다. "기존 시스템이 못 잡는 변동성이 실제로 존재하는가"가 첫 번째 질문이어야 한다.

## 2. 재정의된 아키텍처: 3계층 하이브리드

Critic 지적을 반영하여, 보고서 01의 원래 아키텍처를 보고서 03이 제안한 3계층 구조로 재정의한다.

### 역할 분담

| 계층 | 담당 | 구체적 역할 | 왜 이 계층이 필요한가 |
|------|------|-----------|---------------------|
| **LLM 계층** (판단 + 설명) | 어떤 파라미터를 왜 조정해야 하는지 판단 | KG 질의, 유사 사례 탐색, 원인 가설 생성, 자연어 보고서 작성, 사람에게 설명 | 비정형 지식(FMEA 문서, 엔지니어 노트)을 활용한 추론은 LLM이 가장 잘 함 |
| **전통 ML/물리 모델 계층** (정밀 계산) | 얼마나 조정할지 수치를 계산 | 파라미터 최적화, 시뮬레이션, 예측 모델, R2R 보정 | 수치 정밀도와 결정론적 일관성은 전통 ML이 우위 |
| **온톨로지/KG 계층** (지식 구조 + 안전) | 조정이 안전 범위 내인지 검증, 지식 축적 | 불량-원인-파라미터 관계 저장, 안전 제약 검증, 이력 축적, 도메인 간 지식 이전 | 추론 경로의 감사 추적과 안전 보장은 명시적 지식 구조가 필수 |

### 흐름

```
불량 감지 (SPC/FDC)
    │
    ▼
[온톨로지/KG] ── 관련 지식 탐색: 과거 유사 불량, 원인 후보, 파라미터 관계
    │
    ▼
[LLM 에이전트] ── 원인 가설 순위화, 어떤 파라미터를 조정할지 판단
    │
    ▼
[전통 ML/물리 모델] ── 구체적 수치 계산 (얼마나 조정할지)
    │
    ▼
[온톨로지] ── 안전 범위 검증 (파라미터가 허용 범위 내인지)
    │
    ▼
[LLM 에이전트] ── 사람 읽기용 보고서 생성 (근거, 변경 diff, 영향 범위)
    │
    ▼
[사람 승인] ── 위험도별 자동/수동 분기
    │
    ▼
[설비 전달] ── OPC UA / REST API로 레시피 다운로드
    │
    ▼
[모니터링 + 이력 학습] ── 결과를 KG에 피드백
```

### 기존 시스템과의 통합

이 시스템은 기존 APC/R2R을 **대체하는 것이 아니라 그 위에 올라가는 계층**이다:

```
┌───────────────────────────────────────┐
│  제안 시스템 (지능형 보조 계층)         │  ← 새로 추가
│  온톨로지/KG + LLM 에이전트 + 전통 ML  │
└──────────────────┬────────────────────┘
                   │ 보조/추천
┌──────────────────▼────────────────────┐
│  기존 APC/R2R 시스템                    │  ← 기존 유지
│  PID/R2R 컨트롤러, SPC, FDC            │
└──────────────────┬────────────────────┘
                   │ 제어 명령
┌──────────────────▼────────────────────┐
│  설비 (PLC/DCS)                        │
└───────────────────────────────────────┘
```

> 상세 아키텍처: [01-system-architecture.md](./01-system-architecture.md)

## 3. 이 시스템이 실제로 필요한 곳과 불필요한 곳

### 필요한 곳 (AI가 가치를 더하는 공정)

| 조건 | 왜 AI가 필요한가 | 예시 |
|------|-----------------|------|
| 비예측적 드리프트가 있는 공정 | 정적 R2R 모델이 따라잡지 못하는 변동 | 반도체 증착 (부산물 축적), 화학 반응기 (촉매 열화) |
| 다변량 비선형 상호작용이 큰 공정 | 단일 파라미터 조정만으로 해결 안 됨 | 디스플레이 OLED 증착, 반도체 식각 |
| 공정 간 상호작용 파악이 필요한 경우 | 전(前)공정 변화가 후(後)공정에 미치는 영향 | 식각→CMP→증착 연쇄 영향 |
| 비정형 엔지니어 지식을 활용해야 하는 경우 | 숙련자의 암묵지가 시스템에 반영 안 됨 | FMEA 문서, 유지보수 기록, 트러블슈팅 노트 |

### 불필요한 곳 (기존 시스템이 충분한 공정)

- 안정적이고 선형적인 공정 (PID/R2R이 95-98% 정확도 유지)
- 데이터가 부족한 공정 (AI는 대량의 양질 데이터가 필요)
- 실시간 결정론적 제어가 필요한 곳 (AI의 확률적 특성은 부적합)
- 규제가 극도로 엄격한 공정 (FDA 21 CFR Part 11, EU Annex 22)

### 가장 유효한 시작점

**1순위: 반도체/디스플레이 증착·식각 공정**
- 데이터 인프라 기존 보유, 비예측적 드리프트 실재, 높은 불량 비용
- Tignis가 ML 기반 폐쇄 루프를 이미 상용 운영 중 (CSManTech 2023-2024)

> 상세: [02-manufacturing-reality.md](./02-manufacturing-reality.md) §2

## 4. 기술적 핵심 발견

### 검증된 것 (확신도 높음)

1. **KG 기반 불량-원인 추적이 RAG보다 효과적이다**: 온톨로지 기반 FMEA 그래프 학습이 단순 RAG 대비 F1을 약 2배 향상 (0.267→0.523). 단, 자동차 센서 조립이라는 특정 도메인의 결과이며 다른 도메인에서는 검증 필요 (arXiv:2510.15428).

2. **소형 LLM + KG가 대형 LLM을 이길 수 있다**: ARKNESS 연구에서 3B 파라미터 Llama-3 + KG가 GPT-4o 수준 정확도 달성. 제조 현장의 프라이버시/비용 제약에서 실현 가능한 경로 (arXiv:2506.13026).

3. **폐쇄 루프 AI 제어는 연속 공정에서 성숙 단계**: Imubit이 정유/화학 공장에서 DCS에 직접 명령을 쓰는 수준의 폐쇄 루프를 상용 운영 중.

4. **LLM의 수치 추론은 구조적으로 취약하다**: 수치 변동에 따라 정확도 급감 (ACL 2025). 레시피의 구체적 수치를 LLM에 직접 맡기는 것은 위험하다.

### 유망하지만 검증 부족 (확신도 중간)

5. **LLM 기반 온톨로지 자동 구축**: ARKNESS의 zero-shot KG 구축이 가능성을 보여주었으나, 다양한 도메인과 규모에서의 산업적 검증은 아직 없다.

6. **제조 멀티에이전트 시스템**: 학술 연구와 Bosch 현장 배포가 진행 중이나, 레시피 자동 조정까지 가는 완전한 폐쇄 루프 사례는 아직 없다.

### 전망 수준 (확신도 낮음)

7. **이산 제조에서의 폐쇄 루프 AI 제어**: 연속 공정(화학/정유)과 달리, 이산 제조(반도체 식각, 조립 등)에서의 AI 폐쇄 루프는 아직 초기 단계.

> 상세: [03-kg-llm-process-control.md](./03-kg-llm-process-control.md)

## 5. 현실적 위험과 장벽

### 가장 큰 위험 3가지

1. **과잉 설계**: 온톨로지+LLM+멀티에이전트 전체가 정말 필요한가? 많은 경우 전통 ML/RL만으로 충분할 수 있다. "기존 시스템이 못 잡는 문제가 구체적으로 무엇인가"를 먼저 확인해야 한다.

2. **LLM 환각의 제조 현장 위험**: LLM이 그럴듯하지만 틀린 파라미터를 자신 있게 제시할 수 있다. 사람 승인이 최후 방어선이지만, 승인 피로(알림이 하루 50건이면 형식적 승인)로 무력화될 수 있다.

3. **데이터 인프라 부재**: 대부분 제조 현장은 AI 이전에 센서→MES 데이터 통합부터 해야 한다. 데이터 인프라 없이 AI를 논하는 것은 기초 없이 건물을 올리는 것과 같다.

### 안전/책임 이슈

- AI가 제안하고 사람이 승인한 뒤 불량이 발생하면 책임 소재가 불명확하다
- EU Annex 22(2026 시행)는 고위험 공정에서 결정론적 모델만 허용 — LLM은 본질적으로 비결정론적
- PLC 측 하드 리밋(물리적 안전 한계)은 AI와 무관하게 항상 유지되어야 한다

### Critic이 지적한 누락 사항

- **구축 비용의 구체적 추정이 없다**: Neo4j 라이센스, LLM API 비용, 도메인 전문가 인건비, 운영 인력 비용
- **팀 구성 요구사항이 없다**: 온톨로지 엔지니어 + ML 엔지니어 + OPC UA 전문가 + 제조 도메인 전문가를 한 조직에서 확보하는 것 자체가 도전
- **시스템 수준 실패 모드 분석이 없다**: KG 데이터 오염, 에이전트 무한 루프, 네트워크 단절 폴백, LLM 서비스 장애 시 대응

> 상세: [02-manufacturing-reality.md](./02-manufacturing-reality.md) §3, §5

## 6. 단계적 도입 전략

### Phase 0: 진단 (1개월)

"이 시스템이 정말 필요한가"를 확인하는 단계.

- 대상 공정의 기존 APC/R2R 성능 baseline 측정
- "기존 시스템이 못 잡는 변동성"이 실제로 존재하는지 확인
- 데이터 인프라 성숙도 평가
- **이 단계에서 "AI가 필요 없다"는 결론이 나올 수 있고, 그것도 가치 있는 결과다**

### Phase 1: 읽기 전용 보조 (2-3개월)

AI가 분석만 하고, 레시피에는 손대지 않는 단계.

- 불량 데이터 분석 → 근본원인 가설 제시 (레시피 변경 권한 없음)
- 엔지니어가 AI 제안과 자신의 판단을 비교
- **핵심 지표**: AI 가설의 정확도, 엔지니어 의사결정 시간 단축 여부

### Phase 2: 승인 기반 제안 (3-6개월)

AI가 레시피 조정안을 생성하되, 반드시 사람이 검토·승인.

- 안전 범위 하드코딩으로 위험한 제안 차단
- "AI 제안 vs 엔지니어 판단 vs 실제 결과" 3자 비교 데이터 축적
- **핵심 지표**: 승인된 제안의 불량률 개선 효과, 거부 사유 분석

### Phase 3: 반자동 제어 (6-12개월)

검증된 안전 범위 내에서 자동 조정 허용.

- 범위 밖 조정은 여전히 사람 승인 필수
- 지속적 모니터링 + 이상 시 자동 롤백
- **핵심 지표**: 전체 OEE 개선, 자동 조정 성공률

## 예상 밖 핵심 발견

1. **"이 시스템이 정말 필요한가"가 첫 번째 질문이다**: 기존 R2R이 95-98% 정확도를 유지하는 안정 공정에서는 AI 추가 실익이 미미하다. "비예측적 드리프트가 있는 특정 공정"에서만 이점이 있다.

2. **LLM은 "수치 결정자"가 아니라 "지식 통역사"로 써야 한다**: LLM의 진짜 가치는 비정형 문서(FMEA, 유지보수 기록)를 읽고, 구조화된 지식과 연결하며, 사람에게 설명하는 역할에 있다.

3. **3B 파라미터 소형 모델 + 지식그래프가 현실적 경로다**: 제조 현장의 프라이버시, 비용, 레이턴시 제약을 고려하면, 거대 클라우드 LLM보다 온프레미스 소형 LLM + KG가 더 적합할 수 있다.

4. **폐쇄 루프의 선두 주자는 연속 공정(화학/정유)이고, 이산 제조(반도체/조립)는 아직 초기다**: 시스템 설계 시 타겟 공정 유형에 따라 전략이 달라져야 한다.
