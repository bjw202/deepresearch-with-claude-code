# 온톨로지+LLM 에이전트 레시피 제어 시스템: 제조 현장 현실 검증

> **핵심 질문**: "온톨로지 기반 지식그래프 + LLM 멀티에이전트로 불량 분석 → 레시피 조정 → 설비 전달"이 실제 제조 현장에서 작동할 수 있는가?
>
> **한 줄 결론**: 기술적으로 가능하지만, "어디에, 어떤 범위로, 어떤 순서로" 도입하느냐가 성패를 가른다. 전면 자동화는 위험하고, 기존 APC/R2R 위에 "지능형 보조"로 시작하는 것이 현실적이다.

---

## 1. 기존 시스템과의 비교: 이미 돌아가고 있는 것들

제조 현장에는 이미 불량 감지 → 원인 분석 → 레시피 조정의 흐름을 담당하는 시스템들이 있다. 제안 시스템이 이들과 어떻게 다른지, 어디서 가치를 더하는지를 명확히 해야 한다.

### 기존 4대 공정 제어 시스템

| 시스템 | 역할 | 강점 | 한계 |
|--------|------|------|------|
| **SPC** (통계적 공정 관리) | 관리도 기반 이상 감지 | 단순하고 검증됨. 25년+ 실적 | 체계적 드리프트(도구 마모 등) 감지 못함. 사후 대응 |
| **FDC** (불량 감지 분류) | 센서 데이터로 실시간 이상 감지 | 장비 상태 모니터링에 강함 | 센서 이상은 감지하지만 "왜?"에 대한 답은 약함 |
| **R2R** (Run-to-Run 제어) | 런 간 레시피 파라미터 자동 조정 | 피드백/피드포워드 기반 정밀 보정 | 정적 모델 기반이라 비선형 드리프트에 취약. DOE 기반 모델이 실제와 괴리될 수 있음 |
| **APC** (고급 공정 제어) | R2R + FDC 통합, 공장 전체 최적화 | 수율/처리량/스크랩 동시 개선 | 구축 비용 높음. 전체 배포해야 효과. 모델 유지보수 부담 |

**출처**: Semiconductor Digest (2003), Orbit Skyline (2024), Sentient Cloud (2021), APMonitor 강의자료

### 제안 시스템이 기존 대비 추가할 수 있는 가치

| 기존 시스템의 한계 | 온톨로지+LLM이 보완 가능한 부분 | 현실 난이도 |
|-------------------|-------------------------------|------------|
| R2R의 정적 모델이 비예측적 드리프트를 따라잡지 못함 | ML 기반 적응형 모델이 드리프트 패턴 학습 | **중** - Tignis 등이 이미 상용화 |
| FDC는 "무엇이 이상한지"는 알지만 "왜"는 모름 | 온톨로지가 인과관계 지식을 구조화하여 근본원인 추론 | **중~상** - 학술 수준에서 검증 중 |
| 공정 간 상호작용 파악이 어려움 (예: 식각→CMP 영향) | 지식그래프가 공정 간 관계를 명시적으로 표현 | **중** - 도메인 지식 구축이 병목 |
| 숙련 엔지니어의 암묵지가 시스템에 반영 안 됨 | LLM + RAG로 문서/경험 지식 활용 | **중** - 데이터 품질에 크게 의존 |
| 새로운 불량 모드에 대한 대응이 느림 | LLM의 추론 능력으로 유사 사례 기반 가설 생성 | **상** - 할루시네이션 위험 |

### AI가 불필요한 영역 (기존 시스템이 이미 충분한 곳)

- **안정적이고 선형적인 공정**: PID/R2R이 95-98% 정확도를 유지하는 공정에서는 AI가 불필요한 복잡성만 추가한다.
- **데이터가 부족한 공정**: AI는 대량의 양질 데이터가 필요하다. 데이터가 적으면 전통적 방법이 더 신뢰성 있다.
- **실시간 결정론적 제어가 필요한 곳**: AI의 확률적 특성은 보장된 응답 시간이 필요한 곳에 부적합하다.
- **기본 CNC 가공, 재고 관리 등 루틴 작업**: 전통적 최적화가 충분하다.

> **의사결정 포인트**: "우리 공정이 R2R만으로 못 잡는 변동성이 있는가?" → 없으면 AI 도입 실익이 낮다.

## 2. 제조 도메인별 적용 가능성

"레시피 조정"의 의미와 난이도는 도메인마다 완전히 다르다.

### 도메인별 적합도 평가

| 도메인 | 레시피의 의미 | AI 적용 가능성 | 핵심 이유 |
|--------|-------------|--------------|----------|
| **반도체** | 식각 가스 유량, RF 파워, 증착 온도/시간 등 수십 개 파라미터 | **높음** | 데이터 풍부, 변동성 큼, 기존 APC 인프라 위에 AI 추가 용이. Tignis, Lam Research 등이 이미 상용 배포 |
| **디스플레이** | OLED 증착 조건, 에칭 파라미터 | **높음** | 반도체와 유사한 구조. 대면적으로 갈수록 균일성 제어가 더 어려워 AI 필요성 증가 |
| **자동차 부품** | 사출 성형 온도/압력, 용접 전류/시간, 도장 조건 | **중간** | 공정 자체는 단순하지만 IATF 16949 인증이 자동 레시피 변경을 까다롭게 만듦 |
| **화학/정유** | 반응기 온도, 촉매 투입량, 압력 | **중간~높음** | 연속 공정이라 R2R보다 MPC(모델 예측 제어)가 주류. Imubit 등이 클로즈드 루프 AI 상용화 |
| **제약** | 배합 비율, 건조 온도, 혼합 시간 | **낮음~중간** | FDA 21 CFR Part 11이 레시피 자동 변경에 매우 엄격. 모든 변경이 변경 통제 이벤트 |
| **식품** | 배합, 가열/냉각 조건 | **낮음** | 비교적 단순한 공정. 규제(HACCP)가 자동 변경에 제한적 |
| **아노다이징/표면처리** | 전해액 농도, 전류밀도, 온도, 시간 | **중간** | 변수 적지만 상호작용 복잡. 소규모 사업장이 많아 데이터 인프라 부족 |

### 가장 유효한 적용 영역

**1순위: 반도체/디스플레이 증착·식각 공정**
- 이유: 수십 개 파라미터의 비선형 상호작용, 풍부한 센서 데이터, 높은 불량 비용, 기존 APC 인프라 존재
- 실제 사례: Tignis의 AI/ML 클로즈드 루프 컨트롤러가 증착 공정에서 상용 운영 중. 드리프트 보정에서 기존 정적 APC 대비 우수한 성능 (CSManTech 2023, 2024 발표)

**2순위: 화학/정유 연속 공정**
- 이유: 연속 최적화가 필요하고, 디지털 트윈과 결합 가능
- 실제 사례: Imubit이 정유/화학 공장에서 클로즈드 루프 AI 최적화를 상용 배포. 기존 최적화가 주간/월간 업데이트인 것을 수분 단위로 단축

**3순위: 자동차 부품 가공**
- 이유: 공정 자체는 단순하지만 품질 요구가 높고, 규제가 비교적 유연
- 주의: IATF 인증 환경에서는 변경 통제 절차가 부담

> **관점 확장**: 도메인 선택보다 더 중요한 변수는 "해당 공장의 데이터 인프라 성숙도"일 수 있다. 아무리 좋은 도메인이어도 센서 데이터가 MES에 통합되지 않으면 AI는 작동하지 않는다.

## 3. 안전/책임 이슈: AI가 만든 레시피를 설비에 내릴 때

이것이 시스템의 가장 민감한 부분이다. "사람 승인" 단계가 있다고 해도, 구조적 위험은 여전하다.

### 위험 유형

| 위험 | 구체적 시나리오 | 심각도 |
|------|---------------|-------|
| **장비 손상** | LLM이 허용 범위를 벗어난 RF 파워나 온도를 제안. 엔지니어가 숫자만 보고 승인 | 높음 |
| **제품 불량** | 온톨로지에 없는 새로운 공정 조건에서 잘못된 인과추론 → 대량 불량 | 높음 |
| **인명 안전** | 화학 공정에서 위험한 농도/온도 조합 제안 | 매우 높음 |
| **할루시네이션** | LLM이 그럴듯하지만 틀린 파라미터를 자신 있게 제시. 73% 정확도(복잡한 태스크 기준) | 높음 |
| **연쇄 오류** | 초기 단계의 잘못된 판단이 후속 단계에 전파 | 높음 |

### 규제/인증 요구사항

| 규제 | 요구사항 | AI 레시피 변경에 대한 영향 |
|------|---------|-------------------------|
| **ISO 9001** | QMS 내에서 AI 알고리즘 검증, 변경 사항 문서화 | 모든 자동 레시피 변경은 문서화된 프로세스 필요 |
| **IATF 16949** (자동차) | ISO 9001 + 자동차 고유 요건. 위험 기반 수명주기 관리 | 레시피 변경은 사전 통제가 필요. 자동 변경에 까다로움 |
| **FDA 21 CFR Part 11** (의약품) | 전자 기록/서명 검증, 감사 추적, 데이터 무결성 | 자동 레시피 변경은 "변경 통제 이벤트"로 취급. 매우 엄격 |
| **EU Annex 22** (2026 시행) | GMP 환경에서 AI 모델은 결정론적이어야 하고, 지속적 성능 테스트 필요 | 고위험 공정에서는 결정론적 모델만 허용. LLM은 본질적으로 비결정론적 |
| **SEMI 표준** (반도체) | 장비 통신 규격(SECS/GEM 등) | 레시피 전달 인터페이스 표준 준수 필요 |

### 책임 소재 문제

**"AI가 제안하고 사람이 승인했는데 불량이 나면, 누구 책임인가?"**

이 질문에 대한 명확한 법적 프레임워크는 아직 없다. 현실적으로:
- AI 제안을 승인한 엔지니어에게 책임이 간다
- 하지만 "AI가 근거를 제시했고 엔지니어가 판단할 시간이 부족했다"면?
- 회사 차원에서 AI 시스템의 검증 절차가 충분했는지가 쟁점

> **수치 투명성**: LLM의 복잡한 태스크 정확도는 약 73% (로봇 코드 생성 기준, arXiv 연구). 이 수치는 제조 레시피처럼 더 제약이 있는 도메인에서는 다를 수 있지만, "LLM 출력을 무조건 신뢰하면 안 된다"는 점은 명확하다.

## 4. 실제 현장 사례

### 이미 작동하고 있는 유사 시스템들

**1. Tignis - 반도체 증착 AI 클로즈드 루프 제어**
- 증착 공정에서 ML 기반 R2R 제어기를 실제 팹에 배포
- 바이프로덕트 축적으로 인한 드리프트를 기존 정적 APC보다 잘 보정
- 핵심 학습: 스코프 정의, 데이터셋 관리, 팹과의 모델 검증이 성공의 핵심
- **주의**: 온톨로지/LLM은 사용하지 않음. 전통적 ML/RL 기반

**2. Lam Research - 베이지안 최적화 + 인간 협업 레시피 생성**
- MFL(Model Feedback Learning) 방식으로 5회 반복만에 올바른 레시피 도출
- 비교: Lam 자체 방법 45회, 시니어 엔지니어 수동 84회
- **주의**: LLM이 아닌 베이지안 최적화 기반

**3. 삼성전자 - AI 에이전트 기반 스마트 팩토리 전환 (2026 발표)**
- 디지털 트윈 시뮬레이션 + 품질/생산/물류 전담 AI 에이전트 배포 계획
- NVIDIA와 협력하여 팹의 실시간 디지털 트윈 구축
- 데이터 기반 분석과 사전 검증을 통한 품질 향상 목표
- **상태**: 발표 단계. 실제 레시피 자동 조정까지 가는지는 미확인

**4. Intel - 인라인 AI 배포**
- 인라인 결함 감지, 다변량 공정 제어, 자동 웨이퍼 맵 패턴 감지/분류
- 빠른 근본원인 분석(RCA), 아웃라이어 감지
- **상태**: 프로덕션 배포 중이나, 자율적 레시피 변경이 아닌 엔지니어 보조 수준

**5. Siemens - PCB 생산 라인 최적화**
- 40,000개 생산 파라미터를 분석하여 X-ray 검사를 30% 절감
- AI로 어떤 보드가 검사 필요한지 식별
- **상태**: 레시피 조정이 아닌 검사 최적화

**6. arXiv 2024 - OPC 레시피 개발에 LLM 에이전트 적용**
- 반도체 OPC(Optical Proximity Correction) 레시피를 LLM이 생성
- RL 최적화와 비슷한 성능 (EPE 8-19% 감소)
- LLM 생성 레시피는 추가 최적화 시간 없이 즉시 적용 가능
- **상태**: 연구 단계. 실제 팹 배포 여부 미확인

### 패턴 분석

| 구분 | 실제 배포 사례 | 제안 시스템과의 차이 |
|------|-------------|-------------------|
| ML/RL 기반 제어 | Tignis, Lam Research, Imubit | 온톨로지/LLM 없이 전통 ML로 충분한 성과 |
| AI 에이전트 | 삼성(계획), OpenSSA(연구) | LLM 에이전트이지만 레시피 자동 변경까지는 아직 |
| LLM 레시피 생성 | arXiv OPC 연구 | 가장 유사하지만 연구 단계 |

> **핵심 발견**: 실제 현장에서 "불량 감지 → AI 분석 → 레시피 자동 조정 → 설비 전달"의 완전한 클로즈드 루프를 LLM+온톨로지로 돌리는 사례는 **아직 없다**. 가장 가까운 것은 ML/RL 기반 시스템이고, LLM은 보조 역할(문서 분석, 근본원인 추론 보조)에 머물러 있다.

## 5. 반대 관점: 이 시스템이 실패할 수 있는 이유

### 5.1 기존 APC/R2R이 더 나은 경우

- 안정적 공정에서 기존 R2R은 95-98% 정확도 유지. AI를 추가해도 개선폭이 미미할 수 있다
- 기존 시스템은 결정론적이고 예측 가능. AI는 확률적이고 블랙박스
- R2R 컨트롤러는 수십 년간 검증된 반면, LLM 기반 시스템은 검증 이력이 없다
- **반증 탐색 결과**: Tignis(CSManTech 2024)는 "AI/ML R2R이 특정 클래스의 공정에서 기존 APC보다 우수하다"고 보고. 즉, 모든 공정이 아닌 **특정 공정**(비예측적 드리프트가 있는)에서만 이점 있음

### 5.2 데이터 품질 문제

- 65%의 제조업체가 파편화된 레거시 시스템을 사용 중
- 센서 데이터가 MES에 통합되지 않은 공장이 대다수
- 온톨로지 구축에 필요한 도메인 지식 정형화 자체가 거대한 작업
- 데이터 드리프트로 인해 ML 모델의 주기적 재훈련이 필요하고, 이것 자체가 운영 부담

### 5.3 현장 엔지니어의 수용성

- 94%의 제조업체가 AI 인력 부족을 겪고 있음
- 54%의 작업자가 AI 관련 역량 강화 필요
- 엔지니어들은 AI를 "경영진의 유행"으로 인식할 수 있음 (변화 저항은 Kubler-Ross 모델을 따름)
- **"AI가 만든 레시피"에 대한 불신은 합리적이다**: LLM의 할루시네이션 위험이 실재하고, 틀렸을 때의 비용이 크기 때문

### 5.4 온톨로지의 구축/유지 비용

- 제조 도메인의 온톨로지 구축은 도메인 전문가 + 지식공학자의 협업이 필요
- 공정이 바뀔 때마다 온톨로지도 업데이트해야 함
- FMEA 문서 → 지식그래프 변환의 품질이 LLM 추출에 의존할 경우, 추출 오류가 지식 기반 자체를 오염시킴

### 5.5 LLM의 본질적 한계

- **비결정론적**: 같은 입력에 다른 출력 가능. EU Annex 22(2026)는 고위험 공정에서 결정론적 모델만 허용
- **할루시네이션**: 그럴듯하지만 틀린 파라미터를 자신 있게 제시
- **추론 한계**: 다단계 추론에서 정확도 급감. 제조 인과관계는 종종 다단계
- **블랙박스**: "왜 이 레시피를 추천했는가"를 설명하기 어려움 → 엔지니어 신뢰 확보 어려움

> **문제 재정의**: 원래 질문 "이 시스템이 현장에서 도움이 될까?"보다 더 적절한 질문은 **"기존 APC/R2R로 못 잡는 문제가 구체적으로 무엇이고, 그 문제 해결에 LLM+온톨로지가 최선의 도구인가?"**일 수 있다. 많은 경우 전통 ML/RL이 더 적합할 수 있다.

## 6. ROI와 도입 전략

### MVP 범위 제안

전면 클로즈드 루프가 아닌, **단계적 가치 입증** 전략이 필요하다.

#### Phase 0: 진단 (1개월)
- 대상 공정의 기존 APC/R2R 성능 baseline 측정
- "기존 시스템이 못 잡는 변동성"이 실제로 존재하는지 확인
- 데이터 인프라 성숙도 평가 (센서 → MES 연결 상태)
- **이 단계에서 "AI가 필요 없다"는 결론이 나올 수 있고, 그것도 가치 있는 결과다**

#### Phase 1: 읽기 전용 보조 (2-3개월, 예산의 10-20%)
- AI가 불량 데이터를 분석하고 근본원인 가설을 제시 (레시피 변경 권한 없음)
- 엔지니어가 AI 제안과 자신의 판단을 비교
- 온톨로지 기반 지식 탐색이 실제로 유용한지 검증
- **핵심 지표**: AI 가설의 정확도, 엔지니어의 의사결정 시간 단축 여부

#### Phase 2: 승인 기반 제안 (3-6개월, 예산의 30-40%)
- AI가 레시피 조정안을 생성하되, 반드시 엔지니어가 검토/승인
- 안전 범위(허용 파라미터 범위) 하드코딩으로 위험한 제안 차단
- "AI 제안 vs 엔지니어 판단 vs 실제 결과" 3자 비교 데이터 축적
- **핵심 지표**: 승인된 제안의 불량률 개선 효과, 거부 사유 분석

#### Phase 3: 반자동 제어 (6-12개월, 예산의 40-60%)
- 검증된 "안전한" 파라미터 범위 내에서 자동 조정 허용
- 범위 밖 조정은 여전히 사람 승인 필요
- 지속적 모니터링 + 이상 시 자동 롤백
- **핵심 지표**: 전체 OEE 개선, 자동 조정 성공률

### ROI 기대치

| 벤치마크 | 수치 | 출처 | 주의사항 |
|---------|------|------|---------|
| 예측 정비 ROI | 300-500% | McKinsey Industry 4.0 보고서 | [인접 도메인: 전반적 제조 AI] 레시피 제어와 직접 비교 어려움 |
| 품질 관리 ROI | 200-300% | 다수 산업 보고서 종합 | 결함 감소 60%, 재작업 감소 기준 |
| AI 공정 제어 ROI | 171% (18개월) | Imubit 사례 | [특정 도메인: 정유/화학] 반도체에 직접 적용 시 차이 가능 |
| 평균 ROI 실현 기간 | 12-36개월 | 산업 종합 | MVP 단계에서 3개월 내 초기 ROI 입증 권장 |

> **이 수치가 틀릴 수 있는 조건**: 위 ROI는 "성공적으로 배포된 사례"의 수치다. 실패/중단된 프로젝트는 통계에 포함되지 않는 생존자 편향이 있다. 실제 AI 프로젝트의 87%가 PoC를 넘지 못한다는 보고도 있다.

### 추천 시작점

**가장 현실적인 MVP**: 반도체/디스플레이의 증착 또는 식각 공정에서, 기존 R2R 컨트롤러의 "보조" 역할로 시작

이유:
1. 데이터 인프라가 이미 있음 (센서, MES, 메트롤로지)
2. 비예측적 드리프트 문제가 실재함 (Tignis 등이 검증)
3. 기존 APC 인프라 위에 추가하는 구조라 리스크가 낮음
4. 불량 비용이 높아 ROI 입증이 쉬움

## 7. 종합 판단

### 이 시스템은 "되나?"

**조건부 YES.** 단, 제안된 형태 그대로는 아니다.

| 제안 시스템의 구성요소 | 현실적 가능성 | 근거 |
|---------------------|------------|------|
| 온톨로지 기반 지식 탐색 | **가능하지만 ROI 불확실** | 학술 연구(arXiv 2510.15428)에서 FMEA→KG 변환 검증. 하지만 구축/유지 비용 대비 가치가 전통 ML 대비 우월한지 미검증 |
| LLM 에이전트의 원인 분석 | **보조 역할로 가능** | 근본원인 가설 생성에 유용. 단, 할루시네이션 위험으로 단독 판단 불가 |
| 자동 레시피 조정안 생성 | **제한적으로 가능** | 안전 범위 내 파라미터 조정은 가능. 범위 밖은 사람 필수. LLM보다 전통 ML/RL이 더 적합할 수 있음 |
| 사람 승인 후 설비 전달 | **필수이자 현실적** | 현재 규제 환경에서 완전 자동화는 불가. 사람 승인은 약점이 아니라 필수 안전장치 |
| 폐쇄 루프 | **장기적으로 가능** | 검증된 안전 범위 내에서 반자동 클로즈드 루프는 Tignis, Imubit이 이미 구현 |

### 가장 큰 위험 3가지

1. **과잉 설계**: 온톨로지+LLM+멀티에이전트 전체가 필요한지, 전통 ML/RL만으로 충분하지 않은지를 먼저 검증해야 한다
2. **할루시네이션**: LLM이 생성한 레시피의 신뢰성 문제는 "사람 승인"만으로 완전히 해결되지 않는다 (승인 피로, 숫자만 보는 형식적 승인)
3. **데이터 인프라 부재**: 대부분의 제조 현장은 AI 이전에 데이터 통합부터 해야 한다

### 후속 탐색 질문

1. **온톨로지 vs. 벡터 DB(RAG)**: 구조화된 온톨로지가 단순한 RAG 기반 문서 검색 대비 실질적 우위가 있는가? 유지보수 비용 대비 정확도 이득은?
2. **LLM vs. 전통 ML의 경계**: 어떤 유형의 불량 분석에서 LLM이 전통 ML 대비 명확한 우위를 보이는가? (구조화된 수치 데이터 vs. 비구조화된 엔지니어 노트)
3. **승인 피로(Approval Fatigue)**: AI 제안이 빈번할 때 엔지니어가 형식적으로 승인하는 문제를 어떻게 방지할 것인가?
