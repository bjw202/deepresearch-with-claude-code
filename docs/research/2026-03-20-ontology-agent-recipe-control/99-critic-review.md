## 1. 도메인 적용성

### 1.1 인접 도메인 연구의 무비판적 적용 사례

**보고서 01 (아키텍처)**:
- arXiv 2510.15428의 FMEA KG 연구(F1@20 = 0.523)를 핵심 근거로 사용하는데, 보고서 자체에서도 "자동차 센서 조립"이라는 특정 도메인임을 언급한다. 그러나 이 수치를 시스템 전체의 KG 파이프라인 설계 근거로 사용하면서, **다른 도메인(반도체, 화학 등)에서의 FMEA 문서 구조 차이가 성능에 미치는 영향**은 논의하지 않았다. 자동차 부품의 FMEA는 비교적 표준화되어 있지만, 반도체 팹의 FMEA는 장비 벤더별로 포맷이 다르고 비정형 텍스트 비중이 높아 LLM 추출 성능이 달라질 가능성이 크다.

- Binghamton 대학의 PCB 리플로우 "적합도 97%" 수치를 레퍼런스로 인용하면서 [인접 도메인: PCB SMT 조립] 태깅을 한 점은 적절하다. 다만 이 수치는 **단일 품질 지표(열 프로파일)**에 대한 것이며, 제안 시스템이 다루는 **다변량 불량 분석과는 근본적으로 다른 문제**임을 더 강조할 필요가 있다.

**보고서 03 (최신 연구)**:
- GraphRAG의 "관계 추론으로 80%+" 정확도를 인용하면서 출처가 articsledge.com, HCLTech로 표기되어 있다. 이는 학술 논문이 아닌 **기업 블로그/마케팅 자료**에서 온 수치다. 보고서에서 "제조 도메인 특화 데이터에서의 성능은 별도 검증 필요"라고 주의를 달았지만, 이 수치 자체의 출처 신뢰도가 낮다는 점은 명시하지 않았다.

- Imubit의 성과 수치(처리량 10-30% 증가, 가동중단 30-50% 감소)를 McKinsey 보고서 인용으로 표기했는데, 보고서 자체에서 "Imubit 단독 성과가 아닌 AI 적용 제조업 전반의 범위"라고 밝힌 점은 정직하다. 하지만 이 수치를 **연속 공정(화학/정유) 맥락에서 나온 것인데 이산 제조에도 적용할 수 있다는 인상**을 주는 문맥에서 사용하고 있어 주의가 필요하다.

### 1.2 도메인 적용성이 잘 처리된 부분

- **보고서 02 (현장 현실)**의 도메인별 적합도 평가표(섹션 2)는 가장 정직하고 유용한 분석이다. 반도체부터 식품까지 7개 도메인의 적용 가능성을 구체적 이유와 함께 평가했고, 규제 환경(IATF, FDA, HACCP)의 영향도 명시했다.
- 보고서 02가 "AI가 불필요한 영역"을 명시적으로 정의한 점은 높이 평가한다. 이런 negative finding이 의사결정에 가장 유용한 정보다.

## 3. 서브에이전트 간 상충점

### 3.1 LLM의 역할 범위에 대한 근본적 긴장

**가장 큰 상충**: 보고서 01은 LLM을 시스템의 핵심 의사결정 엔진으로 설계하고, 보고서 02와 03은 LLM의 한계를 반복적으로 경고한다.

- **보고서 01**: "분석 에이전트"가 KG를 질의하여 원인 후보를 순위화하고, "레시피 생성 에이전트"가 파라미터 조정안을 생성한다. 이 구조에서 LLM은 **수치적 의사결정의 주체**다.
- **보고서 03**: "LLM은 수치 추론에 취약하다" (arXiv:2509.06332, ACL 2025). "구체적인 수치는 물리 모델이나 전통 ML이 계산하는 구조가 필수적이다."
- **보고서 02**: "LLM의 복잡한 태스크 정확도는 약 73%." "LLM 출력을 무조건 신뢰하면 안 된다."

**이 상충은 해결되지 않았다.** 보고서 01의 아키텍처 다이어그램에는 "전통 ML/물리 모델" 계층이 명시적으로 존재하지 않는다. 보고서 03이 제안한 "3계층 하이브리드 아키텍처(LLM=판단+설명, 온톨로지=지식+안전, 전통 ML=수치 계산)"를 보고서 01의 아키텍처에 반영하지 않으면, 설계와 현실 사이에 심각한 간극이 생긴다.

### 3.2 온톨로지 구축 비용에 대한 온도 차이

- **보고서 01**: KG 초기 데이터 품질을 "숨은 변수"로 언급하면서도, FMEA 기반 LLM 추출을 해결책으로 제시하며 비교적 낙관적이다.
- **보고서 02**: "온톨로지 구축에 필요한 도메인 지식 정형화 자체가 거대한 작업"이며 "추출 오류가 지식 기반 자체를 오염시킴"이라고 경고한다.
- **보고서 03**: ARKNESS의 zero-shot KG 구축을 "온톨로지 자동 구축이 현실화되고 있다"로 평가하면서도, "복잡한 공리 구조와 논리 규칙에서는 아직 한계"라고 균형을 잡는다.

**판단**: 보고서 02의 우려가 더 현실적이다. ARKNESS(보고서 03)는 CNC 가공이라는 비교적 잘 정의된 도메인의 155개 질문으로 검증했고, 실제 제조 현장의 FMEA 문서는 형식도 품질도 천차만별이다. LLM 기반 자동 구축이 "비용을 낮추는" 것은 맞지만 "문제를 해결하는" 것은 아직 아니다.

### 3.3 기존 시스템 대체 vs 보완에 대한 불일치

- **보고서 01**: 독립적인 풀스택 시스템으로 설계. 기존 APC/R2R과의 통합 접점이 불명확하다.
- **보고서 02**: "기존 APC/R2R 위에 '지능형 보조'로 시작하는 것이 현실적"이라고 명확히 주장한다.
- **보고서 03**: "KG + LLM은 기존 ML/DL을 대체하는 것이 아니라 보완하는 것"이라고 결론짓는다.

**판단**: 보고서 02, 03이 일관되고 보고서 01이 이 관점을 충분히 반영하지 못했다. 아키텍처 설계에서 기존 APC/R2R과의 통합 인터페이스를 명시해야 한다.

## 4. 누락 관점

### 4.1 비용 분석의 부재

세 보고서 모두 **구축 비용의 구체적 추정**이 없다. 다음이 필요하다:

- Neo4j 라이센스 비용 (AuraDB vs Self-hosted)
- LLM API 호출 비용 (건당 분석 비용, 월간 운영 비용 추정)
- 온톨로지 구축에 필요한 도메인 전문가 인건비 (man-month 추정)
- Edge Gateway 하드웨어 비용
- LangSmith, Grafana 등 모니터링 도구 비용
- 운영 인력 (시스템 유지보수, 온톨로지 업데이트)

보고서 02가 ROI를 논의했지만, "입력 비용" 없이 ROI를 말하는 것은 의미가 제한적이다.

### 4.2 실행 가능성의 구체성 부족

- **보고서 01**의 기술 스택 추천은 포괄적이지만, **팀 구성과 역량 요구사항**이 없다. 이 시스템을 구축하려면 어떤 역량의 인력이 몇 명 필요한가? 온톨로지 엔지니어, ML 엔지니어, OPC UA 전문가, LLM 애플리케이션 개발자, 제조 도메인 전문가 등의 조합이 필요한데, 이런 인력을 한 조직에서 확보하는 것 자체가 도전이다.

- **보고서 02**의 단계별 도입 전략(Phase 0-3)은 합리적이지만, 각 단계의 **소요 기간과 의사결정 기준**이 모호하다. "Phase 1에서 Phase 2로 넘어가려면 어떤 지표가 어떤 수준이어야 하는가?"에 대한 답이 없다.

### 4.3 실패 모드 분석의 부족

보고서 02가 위험 유형을 나열했지만, **시스템 수준의 실패 모드 분석(FMEA)**이 없다:

- **KG 데이터 오염**: 잘못된 FMEA 추출이 KG에 들어가면 시스템이 체계적으로 틀린 분석을 반복한다. 이를 감지하는 메커니즘은?
- **에이전트 루프**: 분석 에이전트가 confidence < 0.8로 반복 루프를 돌 때, 최대 반복 횟수와 타임아웃 정책이 보고서 01에 없다.
- **승인 피로**: 보고서 02가 문제를 제기했지만 해결책을 제시하지 않았다. 하루에 알림이 50건이면 엔지니어는 형식적으로 승인하게 된다.
- **네트워크 단절**: Edge Gateway와 클라우드 간 연결이 끊겼을 때 폴백 정책이 보고서 01에 없다.
- **LLM 서비스 장애**: 클라우드 LLM API가 다운되면 시스템 전체가 멈추는가? 로컬 폴백 모델이 있는가?

### 4.4 데이터 거버넌스

- 제조 데이터의 **분류와 접근 제어** 정책이 없다. KG에 축적되는 불량-원인 데이터는 기업의 핵심 영업비밀이다.
- **모델 거버넌스**: LLM의 버전 관리, 성능 모니터링, 드리프트 감지, 재학습 정책이 없다.

## 6. 문제 정의 검토

### 6.1 원래 질문의 적절성

세 보고서 모두 원래 질문을 재정의하는 시도를 했고, 이것이 이번 리서치의 가장 큰 가치 중 하나다:

- **보고서 01**: "아키텍처보다 KG의 콜드 스타트와 지속적 학습이 성패를 결정"
- **보고서 02**: "기존 APC/R2R로 못 잡는 문제가 구체적으로 무엇이고, LLM+온톨로지가 최선의 도구인가?"
- **보고서 03**: "LLM/온톨로지/전통ML의 3계층 하이브리드 최적 결합 아키텍처는 무엇인가?"

**판단**: 보고서 02의 문제 재정의가 가장 날카롭다. "이 시스템을 어떻게 만들까?"보다 "이 시스템이 진짜 필요한가?"를 먼저 묻는 것이 올바른 순서다. 보고서 01은 "어떻게"에 집중하느라 "왜"와 "정말 필요한가"를 충분히 검토하지 못했다.

### 6.2 누락된 문제 정의 요소

- **대상 공정의 특정**: 세 보고서 모두 "어떤 공정에 적용할 것인가"를 열어놓았다. 보고서 02가 도메인별 적합도를 평가했지만, 아키텍처(보고서 01)는 도메인 불특정 설계다. 실제 구현에서는 대상 공정이 특정되어야 기술 선택이 구체화된다.
- **성공 기준의 정의**: "이 시스템이 성공했다"의 정량적 기준이 없다. 불량률 X% 감소? 엔지니어 의사결정 시간 Y분 단축? OEE Z% 향상? 성공 기준 없이 시스템을 설계하면 평가가 불가능하다.

## 종합 평가

### 각 보고서의 핵심 강점

| 보고서 | 핵심 강점 |
|--------|----------|
| **01 (아키텍처)** | 전체 시스템의 구체적 설계를 제시. 상태 스키마, 에이전트 그래프, 설비 연동 프로토콜까지 실제 구현에 참고할 수 있는 수준의 상세도. 승인 UI 목업이 특히 유용 |
| **02 (현장 현실)** | 가장 정직한 보고서. 기존 시스템과의 비교, "AI가 불필요한 영역" 정의, 실패 가능성 분석이 의사결정에 실질적으로 유용. 단계별 도입 전략이 현실적 |
| **03 (최신 연구)** | 학술 문헌 조사의 폭과 깊이가 뛰어남. 근거 신뢰도 매트릭스와 역방향 의사결정 가이드가 체계적. "소형 LLM + KG가 대형 LLM을 이길 수 있다"는 발견이 실용적으로 중요 |

### 통합 시 반드시 해결해야 할 3가지

1. **아키텍처에 전통 ML/물리 모델 계층 추가**: 보고서 01의 에이전트 구조에 보고서 03이 제안한 "3계층 하이브리드"를 반영해야 한다. 레시피 생성 에이전트가 LLM만으로 수치를 결정하는 구조는 위험하다. LLM이 "어떤 파라미터를 조정할지" 판단하고, 전통 ML/물리 모델이 "얼마나 조정할지" 계산하며, 온톨로지가 "안전 범위 내인지" 검증하는 역할 분담이 필요하다.

2. **기존 APC/R2R과의 통합 설계**: 보고서 01의 아키텍처에 기존 공정 제어 시스템과의 인터페이스를 명시해야 한다. Greenfield(처음부터 구축)가 아니라 Brownfield(기존 시스템 위에 추가) 전제의 설계가 현실적이다.

3. **구축 비용과 성공 기준의 정량화**: ROI 논의가 있으려면 비용 추정이 선행되어야 하고, 성공 여부를 판단하려면 정량적 성공 기준이 사전에 정의되어야 한다.

### 보고서 간 신뢰도 서열

의사결정 근거로서의 신뢰도:

**보고서 02 > 보고서 03 > 보고서 01**

- 보고서 02가 가장 보수적이고 현실적이다. 과도한 기대를 교정하는 데 가장 유용하다.
- 보고서 03은 학술적 근거가 탄탄하고 체계적이지만, 일부 수치의 출처 신뢰도에 주의가 필요하다.
- 보고서 01은 구현 참고 자료로서 가치가 높지만, 현실 제약에 대한 고려가 상대적으로 약하다. 보고서 02, 03의 피드백을 반영하여 보강해야 한다.
