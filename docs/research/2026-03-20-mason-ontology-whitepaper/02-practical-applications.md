> 조사일: 2026-03-20

> 초점: "그래서 뭘 할 수 있는가" — 실무자 관점의 활용 가능성

## 1. MASON 개요: 왜 이 온톨로지가 중요한가

MASON(MAnufacturing's Semantics ONtology)은 2006년 Lemaignan 등이 제안한 **제조 도메인 상위 온톨로지(upper ontology)**다. 핵심 구조는 세 가지 축으로 단순하다:

| 축 | 클래스 | 설명 |
|---|---|---|
| **Entity** | 제품(Product) | 무엇을 만드는가 |
| **Operation** | 공정(Process) | 어떻게 만드는가 — 가공, 물류, 인적작업, 준비작업 포함 |
| **Resource** | 자원(Resource) | 무엇으로 만드는가 — 공작기계, 공구, 인적자원, 지리적자원 |

270개 이상의 기본 개념(base concept)과 50개 이상의 속성(property)으로 구성되며, OWL(Web Ontology Language)로 구현되어 있다. SourceForge에서 오픈소스로 공개되어 있다.

**핵심 가치**: 제조 현장의 서로 다른 시스템(ERP, MES, CAD/CAM, 에이전트 시스템 등)이 **같은 언어로 데이터를 주고받을 수 있는 공통 의미 체계**를 제공한다.

> **출처**: Lemaignan et al., "MASON: A Proposal For An Ontology Of Manufacturing Domain," IEEE Workshop on Distributed Intelligent Systems, 2006. Semantic Scholar 기준 **353회 인용** — 제조 온톨로지 분야에서 가장 많이 참조되는 기초 연구 중 하나.

## 2. MASON의 실제 적용 사례

### 2.1 자동 원가 추정(Automatic Cost Estimation)

MASON의 최초이자 가장 직접적인 적용 사례다.

- **방법**: CLIPS 기반 전문가 시스템(expert system)에서 MASON 인스턴스를 활용. 제품의 제조 개념을 정의하면, 온톨로지가 필요한 모든 활동(activity)을 자동 인스턴스화하고, 각 활동에 비용을 연결하여 총 원가를 산출한다.
- **작동 원리**: `canBeMachinedBy` (원자재 → 공정), `uses` (공정 → 공구) 등의 관계를 통해 제품 설명만으로 공정 체인과 자원 할당을 자동 추론한다.
- **실무 의의**: 설계 단계에서 수작업 없이 생산 원가를 사전 평가할 수 있다. 이는 DFM(Design for Manufacturing) 의사결정을 가속화한다.

> **수치 투명성**: 원가 추정의 정확도에 대한 구체적 벤치마크 수치는 원 논문에 보고되지 않았다. 이는 초기 개념 검증(proof of concept) 단계의 연구이며, 실제 양산 환경에서의 정확도는 온톨로지의 인스턴스 품질과 비용 규칙의 정밀도에 크게 의존한다.

### 2.2 시맨틱 멀티에이전트 시스템(Semantic-Aware Multi-Agent System)

- **방법**: 제조 체인의 각 주체(기계, 작업자, 물류 등)를 에이전트로 모델링하고, MASON이 에이전트 간 **인지 모델(cognitive model)**을 제공한다.
- **작동 원리**: 에이전트들이 온톨로지를 공유함으로써 자율적으로 지식 기반 추론과 상호작용이 가능하다.
- **관련 프로젝트**: PABADIS(Plant Automation Based on Distributed Industrial Systems) 프로젝트에서 멀티에이전트 기반 생산 제어에 온톨로지를 활용한 선행 연구가 있으며, MASON은 이 흐름의 연장선에서 개발되었다.

> **반증 탐색**: 멀티에이전트 + 온톨로지 조합의 산업 현장 대규모 배포 사례는 문헌에서 확인되지 않았다. 대부분 연구 프로토타입 수준이며, 실제 공장 MES를 완전히 대체한 사례는 반증 미발견.

### 2.3 제조 가능성 분석(Manufacturability Analysis)

MASON을 기반으로 확장된 **MARON(MAnufacturing Restriction ONtology)**이 제조 제약 조건을 구조화하여 활용된다.

- **활용 분야**: 공정 체인 제약 네트워크(PCCN), CAE 환경에서의 DFM 의사결정 지원
- **실제 사례**: 목조 패널 프레임(wood frame assembly) 제조에서 BIM 모델의 한계를 보완하기 위해 MASON 기반 온톨로지를 확장. 제품-공정-자원 세 클래스를 생성하여 필요한 제조 공정을 자동 식별함 (An et al.)
- **한계**: 비용/시간 추정과 조립 순서 설명에는 충분한 상세도가 부족했음

### 2.4 오프사이트 제조 생산 워크플로우(OPW Ontology)

- 벽 패널 생산의 반자동화 생산라인에 대한 **시맨틱 디지털 트윈** 사례
- MASON의 핵심 개념을 인용하되, 건축 도메인의 복잡성과 중복을 피하기 위해 별도의 경량 온톨로지(OPW)를 개발
- 비용, 시간, 탄소 발자국을 평가하여 지속적 개선을 위한 의사결정을 지원
- **실무 시사점**: MASON을 그대로 쓰기보다, 도메인 특화 온톨로지의 **참조 프레임워크**로 활용하는 패턴이 더 일반적

## 3. MASON을 확장한 실제 프로젝트들

### 3.1 ExtruOnt — 압출기 온톨로지

**가장 체계적인 MASON 확장 사례**로, Industry 4.0 시나리오에서 제조 기계를 시맨틱하게 기술하기 위한 온톨로지다.

| 항목 | 내용 |
|---|---|
| **대상 기계** | 압출기(Extruder) |
| **개발 방법론** | NeOn (6-Phase + Merging Phase Waterfall) |
| **모듈 구성** | 부품(components), 공간연결(spatial connections), 특징(features), 3D 표현, 센서 |
| **상위 온톨로지 정렬** | DUL + **MASON** + SAREF4INMA |
| **MASON 활용 방식** | `MASON:Machine-tool`을 `Extruder` 클래스의 상위 클래스로 설정 |
| **실무 협력** | IBDS(스마트 제조 서비스 기업) 전문가와 공동 개발 |

- **MASON 선택 이유**: MaRCO(기계 능력 중심으로 범위 불일치), MSDL(개념 과다), PSL(프로세스 전용) 등 대안을 검토한 뒤, 도메인 적합성/유지보수/수용도/커버리지 기준으로 MASON을 선택
- **실무 활용 가능성**:
  - 초보 작업자가 3D 렌더링을 통해 압출 공정을 학습
  - 온톨로지 기반 Visual Query System으로 비전문가도 기계 데이터를 탐색
  - 부품 설계 결함의 조기 탐지

> **출처**: Ramírez-Durán et al., "ExtruOnt: An ontology for describing a type of manufacturing machine for Industry 4.0 systems," Semantic Web 11(6): 887-909, 2020. 22회 인용.

### 3.2 SAREF4INMA — 스마트 산업/제조 확장

ETSI(유럽통신표준기구)가 주도하는 SAREF(Smart Appliances Reference) 온톨로지의 제조 확장판이다.

- **26개 신규 클래스, 20개 신규 속성** 정의
- 공장 내 생산 장비의 추적(trace-back)과 모니터링에 초점
- 공장 레이아웃 기술 능력 포함
- Platform Industrie 4.0(독일), Smart Industry(네덜란드), Industria 4.0(이탈리아) 등 **유럽 각국의 I4.0 이니셔티브와 정렬**
- MASON을 참조 온톨로지 중 하나로 인용

> **실무 의의**: SAREF4INMA는 MASON보다 더 구체적인 스마트 제조 표준으로, 유럽 시장에서 공장 간 상호운용성을 구현할 때 실질적으로 적용 가능한 수준이다.

### 3.3 I40GO — Industry 4.0 Global Ontology (2025)

2025년 발표된 가장 최신의 통합 온톨로지로, Industry 4.0 지식을 **전역적으로 통합**하려는 시도다.

- **구성**: 시스템 도메인(디바이스, 센서, 액추에이터, 소프트웨어), 제조 도메인(조직, 공급자, 생산공장, 제품)
- **설계 원칙**: 재사용성(reusability)과 사용성(usability)의 균형
- **MASON과의 관계**: I40GO는 MASON의 제품-공정-자원 구조를 포함하면서, IoT/센서/소프트웨어 시스템까지 범위를 확장

### 3.4 CDM-Core — CREMA 프로젝트

- 독일 DFKI에서 개발한 제조 도메인 온톨로지
- MASON을 "Upper Manufacturing" 카테고리로 분류하여 참조
- SSN(Sensor Ontology), GeoNames 등과 통합하여 클라우드 기반 탄력적 제조(Cloud-based Resilient Manufacturing) 지원

## 4. Industry 4.0 / 디지털 트윈 맥락에서 온톨로지의 현재 역할

### 4.1 AAS(Asset Administration Shell) — 사실상의 표준

현재 Industry 4.0에서 가장 강력한 모멘텀을 가진 것은 **AAS(자산 관리 셸)**다.

| 항목 | 내용 |
|---|---|
| **정의** | 물리적 자산의 제조사 독립적 디지털 표현 |
| **주도** | OPC Foundation, ZVEI, VDMA, IDTA |
| **OPC UA 연동** | OPC 30270 (I4AAS) 사양으로 표준화 |
| **서브모델** | AID(Asset Interface Description), CSS(Capability/Skill/Service) 등 |
| **활용** | 시뮬레이션, 실시간 모니터링, 예방 정비, 공정 최적화 |

- AAS는 MASON 같은 전통 온톨로지와 직접적으로 경쟁/보완 관계에 있다.
- AAS는 **기술 중립적 개념**을 OPC UA로 구현하는 방식이며, 온톨로지의 시맨틱 풍부함보다 **산업 현장 배포 가능성**에 초점을 맞춘다.

### 4.2 RAMI 4.0 — 참조 아키텍처

- 독일 Plattform Industrie 4.0이 정의한 3차원 참조 모델
- 제품(Product) ~ 연결된 세계(Connected World)까지 6개 계층(Layer), 라이프사이클, 계층 레벨을 통합
- 전통적 피라미드 구조(ISA-95)를 **네트워크 구조**로 전환
- MASON의 개념(제품, 프로세스, 자원)이 RAMI 4.0의 계층 안에 자연스럽게 매핑됨

### 4.3 현재 위치: MASON은 "참조 프레임워크"

2026년 현재, MASON 자체가 현장에 직접 배포되는 경우는 드물다. 대신:

1. **새로운 도메인 온톨로지 개발 시 참조 구조**로 활용 (ExtruOnt, OPW, CDM-Core 등)
2. **AAS/OPC UA 기반 시스템의 시맨틱 보강**을 위한 배경 지식으로 인용
3. **학술 연구의 벤치마크**로 지속적으로 인용 (353회+)

> **의사결정 연결**: MASON을 직접 배포하려는 시도는 비효율적이다. 대신 MASON의 구조(Entity-Operation-Resource)를 자사 도메인에 맞게 확장하고, AAS/OPC UA와 연동하는 것이 현실적 경로다.

## 5. 제조 온톨로지 생태계의 주요 플레이어와 표준화 동향

### 5.1 주요 온톨로지/표준 비교

| 이름 | 유형 | 범위 | 관리 주체 | 현재 활성도 |
|---|---|---|---|---|
| **MASON** | 상위 온톨로지 | 제품-공정-자원 | 오픈소스 (SourceForge) | 참조용으로 활성 |
| **PSL** | 프로세스 온톨로지 | 공정 계획, 스케줄링 | NIST (ISO 18629) | 안정, 저활성 |
| **AAS** | 디지털 표현 표준 | 자산 전체 라이프사이클 | IDTA, OPC Foundation | **매우 활성** |
| **OPC UA** | 통신/정보 모델 | 산업 통신 전반 | OPC Foundation | **매우 활성** |
| **SAREF4INMA** | 도메인 확장 | 스마트 제조 | ETSI | 활성 |
| **I40GO** | 글로벌 온톨로지 | I4.0 전체 | Mondragon Univ. | 최신 (2025) |
| **MaRCO** | 능력 온톨로지 | 제조 자원 능력 | 학술 | 활성 |
| **AutomationML** | 데이터 교환 포맷 | 엔지니어링 데이터 | AutomationML Consortium | 활성 |

### 5.2 표준화 동향 (2024-2025)

- **W3C OKI(Ontologies and Knowledge Graphs in Industry) Community Group** 신설: ISO 81346 등 산업 표준을 온톨로지로 변환하는 거버넌스 허브 역할
- **ISO/IEC JWG 21**: Smart Manufacturing Reference Model(SMRM) 개발 — 17개 참조 아키텍처 모델을 분석하여 "메타 언어" 구축 중
- **LLMs4OL Challenge** (2024, 2025): LLM을 활용한 온톨로지 자동 학습 대회. GPT-4o 등이 용어 추출, 분류 체계 구축, 관계 발견에서 유망한 성과

## 6. 실제 현장 도입 시 장벽과 한계

### 6.1 기술적 장벽

| 장벽 | 설명 | 심각도 |
|---|---|---|
| **레거시 시스템 통합** | 기존 PLC/SCADA/MES와 OWL 기반 온톨로지의 접점 부재 | 높음 |
| **OWL 표현력 한계** | 제조 규칙을 OWL 제약조건만으로는 충분히 표현 불가. SWRL 등 추가 필요 | 중간 |
| **인스턴스 품질** | 온톨로지 스키마가 좋아도, 인스턴스 데이터의 품질이 낮으면 무의미 | 높음 |
| **유지보수 부담** | 공정 변경, 설비 추가 시 온톨로지를 지속적으로 갱신해야 함 | 높음 |

### 6.2 조직적 장벽

| 장벽 | 설명 |
|---|---|
| **인력 부족** | 온톨로지 엔지니어 + 제조 도메인 전문가를 겸비한 인력이 극히 희소 |
| **ROI 불확실성** | 온톨로지 구축 비용 대비 정량적 이득을 사전에 증명하기 어려움 |
| **문화적 저항** | "이미 돌아가는 시스템을 왜 바꾸나"라는 현장 저항 |
| **SME 자원 한계** | 중소기업은 온톨로지 구축에 필요한 시간/비용/인력을 투입하기 어려움 |

### 6.3 생태계적 장벽

- **온톨로지 파편화**: MASON, PSL, MaRCO, ADACOR 등 수십 개 온톨로지가 각각 다른 관점에서 개발되어, "어떤 것을 써야 하는가"부터 결정이 어려움
- **표준 간 불일치**: AAS와 전통 온톨로지(OWL 기반) 사이의 매핑이 완전하지 않음
- **온톨로지 레지스트리 부재**: 2024년 RWTH Aachen의 리뷰 논문이 제조 온톨로지의 포괄적 레지스트리를 처음 제공 — 그 전까지는 체계적 목록조차 없었음

> **이 수치가 틀릴 수 있는 조건**: Deloitte 2025 설문에서 제조업체의 54%가 "통합 데이터 표준을 적용 중"이라고 답했으나, 이 "표준"이 온톨로지 수준인지 단순 데이터 스키마인지는 구분되지 않음.

## 7. 반대 관점: 온톨로지 없이 해결하는 접근

### 7.1 데이터 중심 접근 (Data-Driven Approach)

Deloitte 2025 설문(제조업 임원 600명) 기준, 현장의 실제 투자 우선순위:

| 기술 | 향후 24개월 투자 계획 비율 |
|---|---|
| 공장 자동화 | 41% |
| 데이터 분석 | 40% |
| 센서 | 34% |
| AI | 29% |
| 클라우드 컴퓨팅 | 29% |
| 비전 시스템 | 28% |
| IIoT | 27% |

**온톨로지는 이 목록에 직접 등장하지 않는다.** 현장의 즉각적 가치는 센서 데이터 → AI/ML 분석 → 예측 정비/품질 관리 파이프라인에서 나온다.

### 7.2 Knowledge Graph vs. Ontology

| 측면 | Knowledge Graph | Ontology |
|---|---|---|
| 추상화 수준 | 인스턴스(구체적 데이터) | 스키마(개념 정의) |
| 유연성 | 높음 (진화 가능) | 안정적 (명확한 정의) |
| 주요 기술 | Property Graph, RDF | OWL, RDF Schema |
| 추론 | 선택적/제한적 | 명시적/핵심 기능 |

**현실적 합의**: 가장 효과적인 시스템은 **온톨로지 + 데이터**를 결합한다. 온톨로지 없는 Knowledge Graph는 의미가 모호해지고, Knowledge Graph 없는 온톨로지는 이론에 머문다.

> Novo Nordisk 사례: 100년 된 제약 기업이 레거시 데이터 사일로를 **Virtual Knowledge Graph(VKG) + 공개 온톨로지**로 통합. 데이터를 온톨로지에 매핑하여 단일 KG로 제시하는 OBDA(Ontology-Based Data Access) 접근을 채택함. [인접 도메인: 제약/바이오] — 제조업과 데이터 사일로 문제는 유사하나, 규제 요구사항과 데이터 성격이 다름.

### 7.3 LLM이 온톨로지를 대체할 수 있는가?

2024-2025년 LLMs4OL 챌린지 결과:

- **LLM이 잘하는 것**: 용어 추출, 분류 체계 구축, 비분류적 관계 발견 — 온톨로지 **구축 과정의 자동화**
- **LLM이 못하는 것**: 사전 정의 없는 새로운 관계 추론, 논리적 일관성 보장, 대규모 온톨로지의 유지보수
- **현실적 위치**: LLM은 온톨로지를 **대체하는 것이 아니라, 구축과 유지보수를 가속화하는 도구**

> **관점 확장**: 온톨로지의 가장 큰 비용은 초기 구축이 아니라 유지보수다. LLM이 관계형 DB 스키마에서 온톨로지를 자동 추출할 수 있다면(2025년 연구), 온톨로지의 유지보수 비용 장벽이 크게 낮아질 수 있다. 이것이 실현되면 "온톨로지 vs. 데이터 중심" 논쟁 자체가 재정의될 가능성이 있다.

## 8. 종합 판단: 실무자를 위한 의사결정 가이드

### 8.1 MASON을 써야 하는 경우

- 제조 도메인 온톨로지를 **새로 설계**할 때 → MASON의 Entity-Operation-Resource 구조를 출발점으로 사용
- **멀티벤더/멀티사이트** 환경에서 공통 어휘가 필요할 때
- 학술 연구에서 제조 온톨로지의 **기준선(baseline)**이 필요할 때

### 8.2 MASON을 쓰지 말아야 하는 경우

- 현장에 **즉시 배포 가능한 솔루션**이 필요할 때 → AAS + OPC UA가 더 현실적
- **IoT/센서 통합**이 주된 목적일 때 → SAREF4INMA 또는 SSN이 더 적합
- 데이터가 이미 잘 구조화되어 있고, **추론보다 분석**이 목적일 때 → Knowledge Graph 직접 구축이 효율적

### 8.3 역방향 의사결정 가이드

| 상황 | 권장 행동 |
|---|---|
| "공장 간 데이터 호환이 안 됨" | AAS 도입 검토 → MASON 구조로 시맨틱 보강 |
| "새 기계/공정 추가 시 정보 연결이 안 됨" | ExtruOnt 패턴으로 도메인 특화 온톨로지 개발 |
| "AI 모델이 문맥 없이 원시 데이터만 봄" | Knowledge Graph + 경량 온톨로지 조합 |
| "온톨로지 구축할 인력이 없음" | LLM 기반 온톨로지 자동 추출 파이프라인 검토 |
| "투자 대비 효과를 증명해야 함" | 원가 추정이나 제조 가능성 분석 등 좁은 범위부터 시작 |

## 9. 후속 탐색 질문

1. **AAS 서브모델과 MASON의 구체적 매핑 방법론**은 무엇인가? — 현재 공식 매핑이 존재하지 않으며, 이것이 실현되면 두 세계의 통합이 가능해진다.
2. **LLM 기반 온톨로지 자동 구축이 제조 도메인에서 실제로 작동하는가?** — LLMs4OL 챌린지는 바이오/의학 도메인 중심이며, 제조 특화 벤치마크는 아직 없다.
3. **중소 제조기업(SME)을 위한 경량 온톨로지 도입 프레임워크**가 존재하는가? — 대기업 중심의 AAS/RAMI 4.0은 SME에게 과도한 복잡성일 수 있다.

## 10. 문제 재정의

원래 질문 "MASON의 실제 활용 사례와 발전 방향"보다 더 적절한 핵심 질문:

> **"2026년 제조 현장에서 시맨틱 상호운용성을 달성하려면, MASON의 개념 구조를 AAS/OPC UA 생태계 안에서 어떻게 재활용할 수 있는가?"**

MASON 자체의 직접 배포보다, MASON이 제시한 **제품-공정-자원 삼각 구조**가 현대 표준(AAS, SAREF, I40GO)에 어떻게 내재화되어 있는지를 이해하고, 이를 자사 맥락에 맞게 구체화하는 것이 실무적으로 더 가치 있는 접근이다.
