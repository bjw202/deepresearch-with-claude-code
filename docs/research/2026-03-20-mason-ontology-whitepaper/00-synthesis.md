# MASON 제조 온톨로지 백서: 종합 보고서

> **대상 독자**: 제조 도메인에 관심 있는 실무자·관리자. 기술 용어는 처음 등장 시 쉬운 설명을 병기한다.
>
> **읽는 순서**: 이 문서를 먼저 읽고, 관심 영역에 따라 상세 보고서(01~03)로 진입한다.

## 한 문장 요약

**MASON은 "공장의 공통 사전"으로 2006년에 제안된 이론적 뼈대이며, 그 자체가 현장에 배포되는 제품은 아니지만, 20년이 지난 지금도 제조 시스템이 서로 대화하기 위한 개념 구조(제품-공정-자원)의 출발점으로 쓰이고 있다.**

---

## 1. MASON이란 무엇인가 — 핵심만

### 제조에 왜 "공통 사전"이 필요한가

공장에는 설계팀, 가공팀, 품질팀, 물류팀이 있다. 각 팀은 서로 다른 시스템(CAD, MES, ERP 등)을 쓴다. 문제는 **같은 단어를 다르게 해석**하는 것이다. "알루미늄 부품"이라는 말을 설계팀은 형상으로 이해하고, 가공팀은 절삭 조건으로 이해하며, 물류팀은 무게로 이해한다. 이 연결고리가 사람의 머릿속에만 있고 시스템에는 없다.

**온톨로지(Ontology)**는 이 문제를 해결하기 위해, 개념들 사이의 관계를 컴퓨터가 이해할 수 있는 형식으로 정리한 "공통 사전"이다.

### MASON의 구조: 제품-공정-자원

MASON(MAnufacturing's Semantics ONtology)은 2006년 프랑스 ENSAM의 Lemaignan 등이 제안한 **제조 도메인 상위 온톨로지**다. 핵심 구조는 놀랍도록 단순하다:

| 개념 | 뜻 | 비유 |
|------|-----|------|
| **Entity (제품)** | 무엇을 만드는가 | 요리의 "메뉴" |
| **Operation (공정)** | 어떻게 만드는가 | 요리의 "레시피" |
| **Resource (자원)** | 무엇으로 만드는가 | 요리의 "도구와 재료" |

이 세 축을 **PPR(Product-Process-Resource) 프레임워크**라고 부른다. "알루미늄 판재(제품)를 CNC 밀링(공정)으로 5축 가공기(자원)를 써서 깎는다"가 PPR의 한 사례다.

MASON은 약 270개의 개념(클래스)과 50개의 관계(프로퍼티)를 정의하고 있으며, OWL-DL이라는 형식 언어로 작성되어 있다. OWL-DL은 "컴퓨터가 읽고, 자동으로 추론할 수 있는 사전 작성법"이라고 이해하면 된다.

**핵심 가치**: 이 관계 덕분에 컴퓨터가 추론할 수 있다. "이 부품은 알루미늄이다" + "알루미늄은 드릴링 가능하다" + "드릴링은 드릴비트를 사용한다" → 시스템이 자동으로 필요한 공정과 공구를 도출한다.

> 상세: [01-theory-and-history.md](./01-theory-and-history.md) §2–3

## 2. MASON은 실제로 어디에 쓰였나

### 직접 적용 사례

MASON이 직접 적용된 사례는 대부분 **연구 프로토타입** 수준이다. 양산 현장에서 MASON을 그대로 배포한 대규모 사례는 확인되지 않았다.

| 사례 | 내용 | 수준 |
|------|------|------|
| **자동 원가 추정** | 제품 정의 → 공정 자동 전개 → 활동별 비용 산출 (CLIPS 전문가 시스템) | 개념 검증 |
| **멀티에이전트 시스템** | 설비·물류·계획 에이전트가 MASON을 공통 언어로 사용 | 연구 프로토타입 |
| **제조 가능성 분석** | MASON 확장판(MARON)으로 DFM 의사결정 지원 | 연구 |
| **건축 패널 생산** | MASON 개념을 참조한 경량 온톨로지(OPW)로 디지털트윈 구축 | 도메인 특화 |

### 참조 프레임워크로서의 활용 (더 일반적)

실무에서 MASON이 더 자주 쓰이는 방식은 **새로운 온톨로지를 만들 때의 참조 구조**다:

- **ExtruOnt** (2020): 압출기 온톨로지. MASON의 `Machine-tool` 클래스를 상위 개념으로 채택. 여러 대안(MaRCO, MSDL, PSL) 중 MASON을 선택한 이유는 범용성과 유지보수 용이성이었다.
- **SAREF4INMA**: 유럽통신표준기구(ETSI)의 스마트 제조 온톨로지. MASON을 참조 온톨로지로 인용.
- **CDM-Core** (CREMA 프로젝트): MASON을 "상위 제조 온톨로지" 카테고리로 분류하여 참조.

### 현재의 현실적 위치

2026년 현재, MASON 자체를 현장에 직접 설치하는 경우는 드물다. 대신:

1. 새 온톨로지 설계 시 **개념 구조의 출발점**으로 사용
2. AAS(자산관리셸)/OPC UA 기반 시스템의 **시맨틱 배경 지식**으로 인용
3. 학술 연구의 **벤치마크**로 지속 인용 (Semantic Scholar 기준 약 350회+, 2026년 3월 현재)

> **Critic 지적사항 반영**: "사실상의 참조 구조"라는 표현은 과도하다. IOF Core는 BFO라는 다른 상위 온톨로지를 채택했으므로, MASON의 PPR이 유일한 표준이 아니다. **"가장 영향력 있는 초기 제안 중 하나"**가 정확한 표현이다.

> 상세: [02-practical-applications.md](./02-practical-applications.md) §2–4

## 3. 제조 온톨로지 생태계의 현재

### MASON만이 아니다: 20년간의 발전

2024년 서베이 논문(Sapel 등)에 따르면 **65개 이상의 제조 온톨로지**가 존재한다. 이들은 세 시대를 거쳐 발전했다:

| 시대 | 기간 | 특징 | 대표 |
|------|------|------|------|
| 1세대: 개별 제안 | 2005~2012 | 개별 연구팀이 자체 온톨로지 제안 | MASON, MSDL, PSL |
| 2세대: 확장·특화 | 2013~2019 | 특정 관점(역량, 센서 등)을 깊이 파는 온톨로지 | MaRCO, ExtruOnt |
| 3세대: 통합·표준화 | 2019~현재 | 파편화 해소, 모듈형 표준 구축 | IOF Core, I40GO |

### 현재 가장 강한 모멘텀: AAS + OPC UA

현장 배포 측면에서 가장 활발한 것은 **AAS(Asset Administration Shell, 자산관리셸)**와 **OPC UA(산업 통신 표준)**다. 이들은 MASON 같은 학술 온톨로지보다 **산업 현장 배포 가능성**에 초점을 맞추고 있다.

| 이름 | 유형 | 현재 활성도 |
|------|------|------------|
| MASON | 학술 상위 온톨로지 | 참조용 |
| AAS | 디지털 표현 표준 | **매우 활성** |
| OPC UA | 산업 통신/정보 모델 | **매우 활성** |
| IOF Core | 표준화 온톨로지 스위트 | 진행 중 |
| SAREF4INMA | 유럽 스마트제조 표준 | 활성 |

> **의사결정 연결**: MASON을 직접 배포하려는 시도는 비효율적이다. 대신 MASON의 PPR 구조를 자사 도메인에 맞게 확장하고, AAS/OPC UA와 연동하는 것이 현실적 경로다.

> 상세: [01-theory-and-history.md](./01-theory-and-history.md) §4–5, [02-practical-applications.md](./02-practical-applications.md) §4–5

## 4. AI 기술과의 접점

### LLM은 온톨로지를 대체하는가?

**현재 기술 수준에서는 대체 관계가 아닌 보완 관계다.** 다만, 이 판단은 AI 기술이 빠르게 변화하고 있으므로 지속적 재검토가 필요하다.

| LLM이 잘하는 것 | 온톨로지가 잘하는 것 |
|-----------------|---------------------|
| 비정형 텍스트에서 패턴 추출 | 명시적, 논리적 추론 |
| 대규모 코퍼스에서 자동 학습 | 전문가 검증된 도메인 지식 |
| 자연어 인터페이스 | 감사 추적(audit trail) 가능 |
| **약점**: 환각, 비결정적 | **약점**: 구축 비용, 유지보수 부담 |

### 실제로 결합이 이루어지고 있는 영역

**확신도 높음 (제조 직접 사례 존재)**:

1. **Knowledge Graph + LLM (GraphRAG)**: Bosch는 제조 유지보수 온톨로지(MMO)를 구축하고, 규칙 기반 + SLM + LLM을 결합한 파이프라인으로 GraphRAG 챗봇을 개발했다. 핵심 교훈: "고품질 KG 구축에는 전문가 참여가 필수"
2. **온톨로지 기반 데이터 통합**: Bosch는 Virtual Knowledge Graph(VKG)로 이기종 제조 데이터를 온톨로지 매핑만으로 통합 질의한다.
3. **의도 기반 스마트 제조**: ISA-95 표준 기반 온톨로지 + fine-tuned LLM(Mistral-7B)으로 자연어 의도를 공정 계획에 매핑하는 연구가 진행 중이다.

**확신도 중간 (범용 연구, 제조 특화 벤치마크 부족)**:

4. **LLM으로 온톨로지 자동 구축**: OntoEKG, TNO 프레임워크 등이 LLM 기반 반자동 온톨로지 구축을 실증했으나, 제조 도메인 특화 벤치마크는 아직 없다.
5. **온톨로지 기반 프롬프트 설계**: 프롬프트에 넣을 도메인 지식을 자유 텍스트가 아닌 온톨로지 구조로 제공하면, 프롬프트의 일관성과 확장성이 개선된다.

**확신도 낮음 (전망 수준)**:

6. **Neurosymbolic AI**: 신경망의 패턴 인식 + 온톨로지의 논리 추론을 결합하는 접근. Gartner, WEF가 핵심 기술로 선정했으나, 제조 특화 정량 비교는 아직 없다.

> **Critic 반영**: 03 보고서가 제시한 사례 대부분은 MASON을 직접 사용한 것이 아니라 자체 제조 온톨로지를 구축한 것이다. "MASON + AI"가 아닌 "제조 온톨로지 일반 + AI"의 접점으로 이해해야 한다.

> 상세: [03-ai-convergence.md](./03-ai-convergence.md)

## 5. 현장 도입의 현실적 장벽

### 기술적 장벽

| 장벽 | 설명 | 심각도 |
|------|------|--------|
| 레거시 통합 | 기존 PLC/SCADA/MES와 OWL 온톨로지의 접점 부재 | 높음 |
| 인스턴스 품질 | 온톨로지 스키마가 좋아도 데이터 품질이 낮으면 무의미 | 높음 |
| 유지보수 부담 | 공정/설비 변경 시 온톨로지 지속 갱신 필요 | 높음 |

### 조직적 장벽

| 장벽 | 설명 |
|------|------|
| 인력 부족 | 온톨로지 엔지니어 + 제조 전문가를 겸비한 인력이 극히 희소 |
| ROI 불확실 | 구축 비용 대비 정량적 이득을 사전에 증명하기 어려움 |
| 문화적 저항 | "이미 돌아가는 시스템을 왜 바꾸나" |

### 누락되기 쉬운 관점 (Critic 보충)

- **비용**: 온톨로지 구축의 구체적 비용(인-월, 금액)에 대한 공개 데이터가 거의 없다. 이는 의사결정의 핵심 변수인데 정보가 불투명하다.
- **거버넌스**: 구축 후 소유권, 변경 권한, 이력 관리를 누가 담당하는지에 대한 산업 합의가 없다.
- **IP/보안**: 제조 온톨로지에는 공정 노하우, 원가 구조 등 민감 정보가 포함될 수 있다. LLM과 결합 시 유출 위험도 고려해야 한다.
- **실패 사례**: 온톨로지 도입이 실패한 구체적 사례가 문헌에 거의 보고되지 않는다. 성공 편향(survivorship bias) 가능성이 있다.
- **MASON OWL 파일의 현재 상태**: SourceForge에 공개되어 있으나, 2006년 이후 공식 업데이트 여부와 현대 OWL 2 호환성은 확인되지 않았다.

## 6. 종합 판단과 의사결정 가이드

### MASON의 위치를 한마디로

MASON은 **"제조의 알파벳"**이다. 알파벳 자체는 도구가 아니지만, 모든 문서와 소통의 기반이 된다. MASON의 PPR 구조(제품-공정-자원)는 이후 거의 모든 제조 온톨로지의 기본 뼈대가 되었다. 하지만 MASON 자체가 2006년 이후 공식적으로 크게 업데이트되지 않았으므로, "MASON을 직접 배포"보다는 "MASON의 개념 구조를 현대 표준(AAS, OPC UA) 안에서 재활용"하는 것이 현실적이다.

### 상황별 권장 행동

| 당신의 상황 | 권장 경로 |
|------------|-----------|
| 공장 간 데이터 호환이 필요하다 | AAS 도입 → MASON 구조로 시맨틱 보강 |
| 새 기계/공정의 정보 연결이 안 된다 | ExtruOnt 패턴으로 도메인 특화 온톨로지 개발 |
| AI 모델이 원시 데이터만 보고 문맥을 모른다 | Knowledge Graph + 경량 온톨로지 조합 |
| 온톨로지 구축할 인력이 없다 | LLM 기반 반자동 온톨로지 구축 파이프라인 검토 |
| 투자 효과를 증명해야 한다 | 원가 추정이나 제조 가능성 분석 등 좁은 범위부터 시작 |

### MASON을 활용하고 싶다면: 5단계

1. **SourceForge에서 OWL 파일 다운로드** → Protege에서 로드하여 클래스 계층 파악
2. **자사 도메인 용어와 매핑** → MASON의 Entity/Operation/Resource에 자사 개념을 대응
3. **필요한 하위 개념 확장** → MASON에 없는 자사 공정/장비 클래스를 추가
4. **지식그래프 구축** → Neo4j 등에 온톨로지 적재, 사내 데이터 인스턴스 연결
5. **LLM/AI 파이프라인 연동** → GraphRAG로 LLM 그라운딩, 또는 프롬프트에 온톨로지 구조 주입

## 예상 밖 핵심 발견

1. **MASON의 가장 큰 유산은 MASON 자체가 아니다**: PPR 구조(제품-공정-자원)가 이후 모든 제조 온톨로지의 기본 문법이 된 것이 진짜 기여. MASON OWL 파일 자체의 직접 사용보다 이 개념적 유산이 더 영향력이 크다.

2. **온톨로지의 가장 큰 적은 "다른 기술"이 아니라 "유지보수 비용"이다**: 기술적으로는 온톨로지가 유효하지만, 현장에서 지속적으로 갱신할 인력과 프로세스가 없어서 방치되는 것이 가장 흔한 실패 패턴으로 추정된다.

3. **LLM은 온톨로지의 가장 큰 적이 아니라 가장 큰 동맹이 될 수 있다**: 온톨로지의 핵심 장벽(구축·유지 비용)을 LLM이 낮출 수 있다면, 오히려 온톨로지 채택이 가속될 수 있다. 다만 이는 아직 전망 수준이다.
