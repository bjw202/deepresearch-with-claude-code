# Critic Review — LLM 멀티에이전트 오케스트레이션 리서치

> Critic 산출물 | 2026-03-20
> 대상 보고서: 01-deepseek-v3-performance.md (R1), 02-multi-agent-frameworks.md (R2), 03-implementation-architecture.md (R3)

## 1. 도메인 적용성

**대상 도메인**: 회사 내부 LLM API(Qwen3-235B-A22B)를 멀티에이전트 시스템으로 확장

### 문제점

1. **벤치마크의 도메인 불일치 (R1)**
   - LiveCodeBench, AIME, MATH-500 등은 알고리즘 문제풀이/수학 벤치마크로, 멀티에이전트 오케스트레이션에서의 실제 성능(시스템 프롬프트 준수, 멀티턴 도구 호출 일관성, 에이전트 간 협업 품질)과 직접적 상관이 낮다. R1은 이 벤치마크들을 "멀티에이전트 적합성"의 근거로 사용하고 있으나, 측정하는 능력이 다르다.
   - SWE-bench Pro가 가장 관련성 높은 벤치마크이지만 21.4%로 낮은 점수를 보이며, 이것이 실제 에이전트 시스템에서의 복잡한 태스크 수행 능력을 더 정확히 반영할 가능성이 있다.

2. **프레임워크 비교의 일반성 (R2)**
   - R2의 프레임워크 비교는 일반적인 멀티에이전트 사용 사례를 기준으로 한다. 자체 호스팅 Qwen3 환경에서의 특수한 제약(function calling 안정성, MoE 특성에 따른 KV cache 관리, thinking mode 토큰 오버헤드)이 프레임워크 선택에 미치는 영향이 충분히 반영되지 않았다.

3. **Claude Code 아키텍처 참조의 한계 (R2, R3)**
   - Claude Code의 멀티에이전트 아키텍처를 참조 모델로 사용하고 있으나, Claude Code는 Claude 모델(상용, 200K 컨텍스트, 높은 instruction following 능력)에 최적화된 설계이다. Qwen3-235B-A22B(IFBench 36.6%, 오픈소스, function calling 프로덕션 검증 부재)에 동일한 패턴을 적용할 때의 성능 차이가 분석되지 않았다.

### 개선 제안
- 벤치마크 기반 평가보다 **실제 멀티에이전트 시나리오(도구 호출 정확도, 멀티턴 일관성, 시스템 프롬프트 준수율)에 대한 자체 벤치마크 설계**를 권장해야 한다.
- Claude Code 패턴 적용 시, Claude와 Qwen3 간의 instruction following 능력 차이가 초래할 수 있는 구체적 실패 모드를 명시해야 한다.

## 2. 수치 근거 검증

### 출처가 명시된 수치 (양호)
- LiveCodeBench Pass@1 65.9% — 출처 명시, 리더보드 직접 참조 (R1)
- SWE-bench Pro 21.4% — 출처 명시 (R1)
- API 비용 비교표 — 각 제공자별 출처 명시 (R1)
- GitHub Stars — agentically.sh 출처 명시 (R2)

### 문제점

1. **출처 불명확 수치 (R1)**
   - "비용 차이 25배" (섹션 5.3): `5개 Researcher x 50K 출력 토큰 = Qwen3 $0.03 vs Claude $0.75` 계산. 이 계산에서 SiliconFlow 가격($0.68/1M 출력)을 사용한 것인지, 어느 제공자 가격인지 불명확. Thinking 모드 미사용 전제도 명시되지 않음.
   - "Thinking 모드에서 토큰 2~5배 증가" (R1 섹션 5.3, R3 섹션 2.2): 이 범위의 출처가 없다. R1은 "2~5배", R3는 "2-3x"로 서로 다른 수치를 제시하며 모두 출처 없음.

2. **프로덕션 준비도 별점 근거 부재 (R2)**
   - LangGraph 5점, CrewAI 3점, AutoGen 3점 등의 별점 평가에 구체적인 기준(채점 기준, 평가 항목 가중치)이 없다. 이 별점은 R2 저자의 종합적 판단이며, 독자가 검증하기 어렵다.

3. **구현 로드맵 기간 추정 (R3)**
   - "MVP 1-2주", "안정화 2-3주" 등의 추정에 팀 규모, 기술 수준, 사전 경험 등의 전제 조건이 없다. 이 수치는 매우 낙관적일 수 있다.

4. **vLLM "43x 처리량 향상" (R3 섹션 6.1)**
   - "정적 배칭 대비 약 43x 처리량 향상 가능"이라고 하나, 이 수치의 출처가 명시되지 않았고, 측정 조건(모델 크기, 배치 크기, 시퀀스 길이)이 없다. Qwen3-235B-A22B에서의 수치인지도 불명확.

### 개선 제안
- Thinking 모드 토큰 오버헤드에 대한 실제 측정 데이터 또는 Qwen 공식 문서의 근거를 보강해야 한다.
- 프로덕션 준비도 별점의 평가 기준을 명시하거나, 별점 대신 항목별 비교표로 대체해야 한다.
- 구현 기간 추정에 전제 조건을 반드시 명시해야 한다.

## 3. 서브에이전트 간 상충점

### 상충 1: Thinking 모드 토큰 오버헤드 수치 불일치

| 보고서 | 주장 | 수치 |
|--------|------|------|
| R1 (섹션 5.3) | Thinking 모드 사용 시 출력 토큰 증가 | **2~5배** |
| R3 (섹션 2.2, 9.3) | Thinking mode 토큰 소비 증가 | **2-3x** |

R1은 상한을 5배로, R3는 3배로 잡고 있다. 비용 추정에 직접 영향을 미치는 수치이므로 통일이 필요하다.

### 상충 2: Function Calling 안정성 평가 불일치

| 보고서 | 평가 | 상세 |
|--------|------|------|
| R1 (섹션 3.1) | "에이전트 도구 사용에 강점", 공식 지원 표 제시 | 긍정적 톤 |
| R2 (섹션 12) | "복잡한 도구 스키마에서의 안정성은 별도 벤치마크 필요", MoE 22B active의 정밀도 우려 | 유보적 톤 |
| R3 (섹션 11.2) | "오픈소스 모델의 function calling은 GPT-4 대비 JSON 파싱 오류가 더 빈번할 수 있다" | 부정적 톤 |

R1은 공식 문서 기반으로 "지원됨"을 강조하는 반면, R2와 R3는 실제 안정성에 대한 우려를 제기한다. "지원 여부"와 "프로덕션 안정성"은 다른 문제인데, 이 구분이 종합 보고서에서 명확히 정리되어야 한다.

### 상충 3: 자체 구현 vs 프레임워크 권장 방향

| 보고서 | 권장 |
|--------|------|
| R2 | 상황별로 프레임워크 선택 권장, 특히 LangGraph를 복잡한 워크플로우에 최적으로 평가 |
| R3 | MVP는 자체 구현으로 시작 → 확장 시 LangGraph로 마이그레이션 권장 |

완전한 모순은 아니지만, R2가 프레임워크의 장점을 강조하는 반면 R3는 자체 구현의 이점(최소 의존성, Qwen3 최적화)을 먼저 내세운다. 실제 의사결정에서 어느 쪽이 더 적합한지에 대한 판단 기준이 명확하지 않다.

### 개선 제안
- 상충점 1: Thinking 모드 오버헤드 수치를 실측하거나, 출처를 명시하여 통일해야 한다.
- 상충점 2: "지원 여부"와 "프로덕션 안정성"을 분리하여 평가하고, 안정성 검증을 위한 구체적 테스트 계획을 제시해야 한다.
- 상충점 3: 팀 규모, 기존 인프라, 타임라인에 따른 명확한 분기 조건을 제시해야 한다.

## 4. 누락 관점

### 4.1 운영 부담 (Operations Overhead)

세 보고서 모두 **구축**에 초점을 맞추고 있으며, 운영 단계에서의 부담이 충분히 다뤄지지 않았다:

- **모델 업데이트 전략**: Qwen3 후속 버전(Instruct-2507, Coder-480B)으로 업그레이드 시 호환성 검증 프로세스
- **프롬프트 유지보수**: 멀티에이전트 시스템에서 에이전트 수 x 프롬프트 수만큼 유지보수 표면적이 증가한다. 프롬프트 버전 관리, 회귀 테스트 방법이 없다.
- **GPU 인프라 운영**: Qwen3-235B-A22B를 FP16으로 서빙하려면 ~470GB GPU 메모리가 필요하다(R3 섹션 6.3). 이 규모의 GPU 클러스터 운영 비용, 장애 대응, 스케일링 전략이 빠져 있다.

### 4.2 실패 모드 분석 부족

R3에서 에러 핸들링을 다루고 있지만, **실패 모드의 체계적 분석**이 없다:

- **무한 도구 호출 루프**: 에이전트가 도구 결과를 이해하지 못하고 같은 도구를 반복 호출하는 경우
- **Hallucinated Tool Call**: 존재하지 않는 도구를 호출하거나, 잘못된 인자 포맷을 반복하는 경우
- **에이전트 간 결과 전달 손실**: 서브에이전트 결과를 메인 컨텍스트에 삽입할 때 truncation으로 핵심 정보가 유실되는 경우 (R3의 `extract_key_result`에서 3000자 제한이 적절한지 검증 없음)
- **Thinking 모드 + Tool Use 충돌**: Thinking 모드에서 `<think>` 블록과 tool call 응답이 혼재되는 파싱 문제

### 4.3 보안 고려사항 부족

R3에서 Docker 샌드박스를 다루지만:

- **Prompt Injection 방어**: R3 섹션 11.2에서 언급만 하고 구체적 방어 전략이 없다
- **데이터 유출**: 서브에이전트가 도구를 통해 접근하는 데이터의 범위 제한 전략이 없다
- **API 키 관리**: 자체 호스팅이라 하더라도, 도구가 외부 API를 호출할 때의 인증 관리 전략이 없다

### 4.4 단일 에이전트 대비 개선 검증 부재

멀티에이전트의 전제는 "단일 에이전트보다 나은 결과를 낸다"인데, 이를 검증하는 방법이 제시되지 않았다. R3 섹션 11.1에서 "A/B 테스트가 필요하다"고 언급하지만 구체적 계획이 없다.

### 개선 제안
- 실패 모드 카탈로그를 작성하고, 각 실패 모드에 대한 탐지/복구 전략을 명시해야 한다.
- GPU 인프라 운영 비용을 API 비용 비교에 포함해야 한다 (자체 호스팅의 진정한 TCO).
- 단일 에이전트 vs 멀티에이전트 비교 실험 설계를 로드맵에 포함해야 한다.

## 5. 확신도 교정

### 과도한 확신 사례

1. **R1 "강력히 추천" (섹션 8, 역할별 추천)**
   > "Researcher (자료 조사): **강력 추천** — 비용 효율적, 충분한 지식/추론 능력"

   프로덕션 멀티에이전트 배포 사례가 없고(R1 섹션 6.3에서 본인이 인정), function calling 안정성이 미검증인 상태에서 "강력 추천"은 근거 대비 과도하다. "조건부 추천 — PoC 검증 후 확대"가 적절하다.

2. **R1 비용 비교 "81% 이상 저렴" (요약)**
   > "API 비용은 Claude Sonnet 4 대비 **81% 이상 저렴**"

   이 수치는 SiliconFlow/Together AI 등 서드파티 API 가격 기준이다. 자체 호스팅 시 GPU 인프라 비용(8xA100 렌탈 $20-40K/월)을 포함하면 비용 구조가 완전히 달라진다. 자체 호스팅과 API 사용을 구분하지 않은 비용 비교는 오해를 유발한다.

3. **R3 "동작하는 멀티에이전트 PoC, 1-2주" (섹션 12)**
   > "MVP (Phase 1): 1-2주 — 동작하는 멀티에이전트 PoC"

   Qwen3의 function calling 안정성이 미검증인 상태에서 1-2주 내에 "동작하는" PoC 달성은 낙관적이다. Function calling 파싱 이슈, Thinking 모드 호환성 문제 해결에 상당 시간이 소요될 수 있다.

4. **R2 LangGraph 프로덕션 준비도 5점 (섹션 1)**
   > "프로덕션 준비도: ⭐⭐⭐⭐⭐"

   LangGraph가 프레임워크 중 가장 프로덕션 지향적인 것은 맞지만, 5점 만점은 "개선 여지 없음"을 암시한다. LangChain 의존성, 추상화 레이어 깊이, 디버깅 복잡성(R2 본인이 단점으로 명시)을 고려하면 4점이 더 적절하다.

### 적절한 확신 수준 (양호)
- R1의 SWE-bench Pro 약점 인정과 반증 탐색
- R2의 "흔한 패턴: CrewAI로 프로토타입 → LangGraph로 프로덕션 마이그레이션"
- R3의 "무조건 멀티에이전트가 아니라 하이브리드 접근이 실용적" (섹션 11.3)

### 개선 제안
- "추천"과 "강력 추천"의 구분 기준을 명시하고, 프로덕션 검증 부재 시 "강력 추천"을 사용하지 않아야 한다.
- 비용 비교 시 자체 호스팅 TCO와 서드파티 API 비용을 별도 시나리오로 분리해야 한다.
- 기간 추정에 불확실성 범위를 포함해야 한다 (예: "1-2주 (optimistic) / 3-5주 (conservative)").

## 6. 문제 정의 검토

### 원래 질문

> "회사 LLM API(Qwen3-235B-A22B)를 Claude Code처럼 멀티에이전트로 확장"

### 문제점

1. **솔루션 고정 (Solution Fixation)**
   - "Claude Code처럼"이라는 참조가 솔루션을 특정 아키텍처 패턴(오케스트레이터-워커, 깊이 1 제한, 격리된 컨텍스트)으로 고정시키고 있다. Claude Code의 아키텍처는 Claude 모델의 높은 instruction following 능력과 200K 컨텍스트를 전제로 설계되었다. IFBench 36.6%인 Qwen3에 동일 패턴을 적용하면 에이전트가 시스템 프롬프트를 일관되게 따르지 못하는 실패가 발생할 수 있다.

2. **"확장"의 정의 불명확**
   - "멀티에이전트로 확장"이 무엇을 의미하는지 불명확하다. 가능한 해석:
     - (a) 코딩 어시스턴트로서의 멀티에이전트 (Claude Code 직접 대체)
     - (b) 범용 업무 자동화 멀티에이전트
     - (c) 특정 도메인(리서치, 데이터 분석 등)에 특화된 멀티에이전트
   - 각 해석에 따라 최적 아키텍처가 달라지는데, 이 구분이 이루어지지 않았다.

3. **핵심 리스크의 은폐**
   - 가장 큰 리스크는 Qwen3-235B-A22B의 **멀티에이전트 환경에서의 실제 안정성**이다. 세 보고서 모두 이 리스크를 인지하고 있으나("프로덕션 안정성 데이터 부재", "별도 벤치마크 필요"), 이것이 프로젝트의 성패를 좌우할 수 있는 핵심 리스크라는 점이 충분히 강조되지 않았다.

### 더 나은 문제 정의 제안

R3의 문제 재정의가 가장 적절하다. 이를 더 발전시키면:

> **"Qwen3-235B-A22B의 function calling 안정성과 instruction following 능력이 멀티에이전트 오케스트레이션에 충분한지 먼저 검증하고, 검증 결과에 따라 (a) 전면 멀티에이전트, (b) 단일 에이전트 + 도구 확장, (c) 하이브리드 접근 중 최적 전략을 결정한다."**

이 정의는:
- 솔루션을 고정하지 않고 검증 기반 의사결정을 유도한다
- 핵심 리스크(function calling 안정성)를 전면에 배치한다
- 멀티에이전트가 유일한 답이 아닐 수 있음을 인정한다
