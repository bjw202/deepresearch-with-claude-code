# 공정 전문가 시스템 — 임원 공유용 브리핑

> 작성일: 2026-03-19 | 대상: AI 비전문 임원 | 성격: 선행 리서치 과제 중간 공유 과제 기간: 2026.01 \~ 2026.12 | 현재 상태: 1단계 핵심 모듈 개발 중 (3\~6월)

---

## 한 줄 요약

\*\*"설비에 문제가 생기면 사람 없이도 AI가 원인을 찾고 레시피를 처방하는 시스템"\*\*을 만들고 있습니다.

---

## 1. 왜 이걸 해야 하는가

공정 불량이 발생하면 원인 파악에 수시간\~수일이 걸리고, 레시피 조정은 설비사 의존도가 높아 즉각 대응이 어렵습니다. 핵심 노하우가 소수 베테랑에게 집중되어 있어 퇴직·부재 시 대응력이 급격히 저하됩니다.

| 문제 | 영향 |
| --- | --- |
| 원인 분석·레시피 조정에 사람 필수 | 야간/주말 대응 공백, 라인 대기 또는 불량 생산 |
| 노하우가 소수 베테랑에 집중 | 인력 이탈 시 10\~20년 경험 소실 |
| 설비사 의존도 높음 | 레시피 변경 리드타임 수일, 높은 비용 |

McKinsey(2024)에 따르면 AI 기반 예측 정비 도입 기업은 다운타임 30\~50% 감소를 보고하고 있습니다. 단, 대기업 성공 사례 위주 수치이며, 우리 환경에서의 효과는 별도 검증이 필요합니다.

---

## 2. 무엇을 만드는가

### 병원에 비유하면

|  | 지금 | 목표 |
| --- | --- | --- |
| 진단 | 의사(엔지니어)가 직접 와서 기억에 의존해 판단 | AI가 과거 기록을 즉시 검색하여 원인 분석 |
| 처방 | 약(레시피)을 수작업으로 조제, 설비사 의존 | 최적 처방전(레시피)을 자동 생성 |
| 사람 역할 | 전 과정을 직접 수행 | 최종 확인과 판단에 집중 |

### 시스템의 3가지 핵심

| 역할 | 비유 | 기술 | 현재 상태 |
| --- | --- | --- | --- |
| **두뇌** — 원인 분석 + 처방 | 여러 전문의가 회의해서 진단 | AI 멀티에이전트 (여러 AI가 역할 분담) | 설계 중 (4\~5월 개발) |
| **기억** — 과거 사례 + 공정 지식 | 전자의무기록 시스템 | 지식 그래프 DB + 유사 사례 검색 | 온톨로지 설계 중 (3\~5월) |
| **손발** — 설비와 통신 | 처방전을 약국에 전송 | API 통신 모듈 | 8\~9월 개발 예정 |

### 왜 일반 DB가 아니라 지식 그래프인가

일반 DB는 개별 데이터를 저장하고 검색합니다. 지식 그래프는 **데이터 간 관계**를 저장하여 "A가 B에 영향을 미치고, B가 C의 원인이 된다" 같은 연쇄 추론이 가능합니다.

| 구분 | 일반 DB | 지식 그래프 |
| --- | --- | --- |
| 비유 | 전화번호부 (이름으로 번호 검색) | 인맥 지도 (누가 누구와 연결되고, 그 관계가 또 무엇에 영향을 주는지) |
| 저장 대상 | 개별 레코드 | 데이터 + 데이터 간 관계 |
| 강점 | 정확한 조건 검색 | 연관 관계 탐색, 원인-결과 추적 |
| 공정 적용 예시 | "온도 250도인 레시피 검색" | "온도 이상 → 어떤 불량 유형 → 어떤 레시피 조정이 효과적이었나"를 한 번에 추적 |

### 온톨로지의 역할 — AI에게 맥락을 주는 구조

**온톨로지**는 공정 파라미터, 레시피, 불량 유형, 소재 등의 **관계 구조를 정의하는 지식 체계**입니다. 병원에 비유하면, 온톨로지는 **진료 가이드라인**(어떤 증상이 나오면 어떤 검사를 하고, 어떤 약을 쓰는지의 관계 체계)이고, 지식 그래프는 이 가이드라인에 따라 구축된 **실제 환자 데이터 네트워크**입니다.

온톨로지가 중요한 이유는 AI에게 \*\*컨텍스트(맥락)\*\*를 제공하기 때문입니다. AI가 공정 문제를 분석할 때, 온톨로지가 "이 파라미터와 관련된 레시피, 소재, 과거 불량 사례"를 자동으로 묶어서 컨텍스트로 전달합니다. AI는 단편적 데이터가 아니라 관련 정보가 연결된 맥락을 받게 되어 판단 정확도가 올라갑니다.

## 3. 지금 어디까지 왔는가 — 솔직한 현황

### 확인된 것 (자신있게 말할 수 있는 것)

| 항목 | 근거 |
| --- | --- |
| AI가 과거 사례를 검색해서 유사 상황을 찾는 것은 **기술적으로 가능** | RAG(검색 증강 생성) 기술이 산업계에서 이미 검증됨 |
| 공정 지식을 구조화하면 AI 판단 정확도가 올라감 | 지식 그래프 + AI 결합 시 수치 오류 22%p 감소 (ARKNESS 논문, 2025) |
| 설비-서버 통신은 구현 가능 | REST API + MCP 프로토콜, 기존 CNC 개발 경험 보유 |
| AI 코딩 도구로 개발 속도 5배 단축 | 1\~2월 준비 단계에서 실증 (Claude Code 기반) |

### 아직 확인해야 하는 것 (솔직하게 말해야 하는 것)

| 항목 | 불확실성 | 언제 확인되나 |
| --- | --- | --- |
| 사내 LLM(Gauss)이 공정 분석에 충분한 성능인가 | **높음** — 범용 LLM과 성능 차이 미측정 | 4월, Wrapper 개발 후 |
| 레시피 추천 정확도 70% 달성 가능한가 | **높음** — 데이터 품질에 크게 의존 | 11\~12월, MVP 테스트 후 |
| 온톨로지(지식 구조화)가 실제로 ROI를 내는가 | **중간** — 학술 연구는 많으나 제조 양산 사례 극히 제한적 | 7\~8월, KG 구축 후 |
| 설비가 AI 레시피를 바로 받아 적용할 수 있는가 | **중간** — 설비별 통신 규격이 다를 수 있음 | 10\~11월, 연동 테스트 후 |
| 자동 대응률 50% 달성 가능한가 | **높음** — 공정 복잡도에 따라 크게 달라짐 | 12월, 성능 측정 후 |
| 파일럿 적용 공정이 미선정 | **높음** — CNC, 화학가공, 레이저, 연삭, 폴리싱 등 후보 다수. 공법마다 데이터 구조, 파라미터 복잡도, 설비 통신 방식이 달라 선정 자체가 중요한 의사결정 | 지속 조사 중. 데이터 품질이 좋고 자동화 효과가 큰 공정을 우선 선정할 계획 |

> **핵심 메시지**: 개별 기술은 검증되었지만, **우리 공정에 맞게 통합했을 때 실제로 작동하는지**는 올해 하반기에 확인됩니다. 이것이 "선행 리서치"인 이유입니다.

## 5. 성공하면 무엇이 달라지는가

### 단기 (올해 검증 목표)

| 지표 | 목표 | 의미 |
| --- | --- | --- |
| 레시피 추천 정확도 | 70% 이상 | AI가 추천한 레시피 10개 중 7개가 바로 쓸 수 있는 수준 |
| 자동 대응률 | 50% 이상 | 설비 이슈 2건 중 1건은 사람 없이 처리 |
| 엔지니어 대응 시간 | 30% 단축 | 수시간 걸리던 분석이 수십 분으로 |

### 중장기 (차년도 이후 기대)

1. **야간/주말 무인 대응 가능** — 24시간 가동 라인의 최대 취약점 해소
2. **설비사 의존도 감소** — 레시피 제어 주도권을 공정팀이 확보
3. **노하우 형식지화** — 베테랑의 경험이 시스템에 축적, 퇴직해도 소실되지 않음
4. **수평 전개** — 한 공정에서 검증되면 동일 구조로 다른 공정에 빠르게 확장

## 6. 리스크와 대비

| 리스크 | 심각도 | 대비책 |
| --- | --- | --- |
| 사내 LLM 성능 부족 | 높음 | 외부 LLM(Claude 등) 병행 사용 옵션 준비. 단, 보안 정책 확인 필요 |
| 공정 데이터 품질 미달 | 높음 | 파일럿 공정 선정 시 데이터 완결성이 높은 공정 우선 선택 |
| 설비 통신 규격 불일치 | 중간 | 표준 프로토콜(REST API) 우선, 설비별 어댑터 개발 |
| 현장 엔지니어 거부감 | 중간 | "대체"가 아닌 "보조" 도구로 포지셔닝. 엔지니어 참여형 검증 |
| 올해 내 검증 미완료 | 낮음 | 12개 태스크를 단계별로 구성하여 중간 성과물 확보 가능 |

## 7. 이 과제를 한마디로

> **"베테랑 엔지니어의 머릿속에만 있던 공정 노하우를, AI가 24시간 활용할 수 있게 만드는 것"**

이 과제는 **완성된 시스템을 만드는 것이 아니라**, 그것이 가능한지를 올해 안에 검증하는 선행 리서치입니다. 12월에 파일럿 결과를 바탕으로 정식 과제화 여부를 판단합니다.
